# V2.2 Dashboard 空状态 / 错误态 / 数据稳定性补强 PRD

## 背景

`moreagent` 的 Dashboard 能力已经分阶段完成：

1. V1.8：JSON 输出
2. V1.9：`report`
3. V2.0：static dashboard
4. V2.0.1：dashboard hardening
5. V2.1：dashboard usability enhancement

当前 V2.1 已放行：

- `npm test`: 66 passed, 0 failed
- `npm run build`: `tsc` 无错误

到 V2.1 为止，Dashboard 已经解决了三件事：

1. **信息可获得**
   - runs 列表
   - run detail
   - workflow report
   - gate / test / review
   - repair
   - merge readiness
   - sessions
   - JSON / debug

2. **基础交互可用**
   - static HTML
   - local run switching
   - `--run`
   - `--limit`
   - `--output`

3. **可读性已有改善**
   - sidebar 状态辨识
   - summary 区增强
   - merge readiness 解释增强
   - workflow / MVP run 降级展示

但是，V2.1 的重点仍然是“正常路径的可读性”，而不是“异常路径的稳定展示”。

V2.2 要解决的问题是：

- 数据为空时怎么展示
- 某个 run 的 detail 缺失时怎么展示
- 某个 JSON 子命令坏掉时页面怎么降级
- 某些字段缺失时如何避免 `undefined/null/[object Object]`
- 哪些情况应该生成 HTML，哪些情况应该直接失败

换句话说，V2.2 的核心不是新增能力，而是补强 **resilience**。

## 为什么要做 resilience

当前 Dashboard 的数据链路是：

1. `status --json`
2. `status --run <id> --json`
3. `report --run <id> --json`
4. `inspect --run <id> --workflow --json`

只要其中任一层出现异常、缺字段、或返回局部 error，用户就可能遇到阅读问题：

1. 页面能打开，但某些区域看不懂
2. 页面能打开，但用户不知道错误是正常降级还是异常
3. 命令直接失败，但用户不知道是哪个子命令坏了
4. 某些字段缺失时显示 `undefined`
5. 局部错误会被误判成整个 run 失败

V2.2 的目标是把这些问题系统化处理。

## 当前 Dashboard 在异常状态下的潜在阅读问题

当前 Dashboard 在异常状态下可能出现这些理解风险：

1. 没有 runs 时，用户不知道是初始化问题、没有执行过，还是 dashboard 自己坏了
2. 某个 run 缺 `status detail`，sidebar 还能显示，但右侧 detail 会变得不完整
3. 某个 run 缺 `report detail`，Summary / Merge / Recommendation 可能无从判断
4. workflow 对 MVP run 不可用，但这是预期降级，不应该渲染成错误
5. workflow 对 full run 异常失败时，应该显示错误，而不是沿用 MVP 降级文案
6. JSON parse 失败时，当前命令可能只看到命令失败，不知道是哪一个子命令的 stdout 非法
7. `report` 缺字段时，页面可能显示 `undefined`
8. merge / worktree 信息缺失时，用户无法判断 blocked 原因
9. sessions 为空但 run 已完成时，用户容易误解为 run 没跑，或 dashboard 数据损坏

## 用户痛点

### 1. 没有 runs 时用户不知道下一步

如果 dashboard 打开后什么都没有，用户会困惑：

- 是没有历史 runs
- 还是 `.moreagent` 没初始化
- 还是 dashboard 坏了

V2.2 必须给出明确解释和下一步建议。

### 2. 某个 run 缺 status detail

sidebar 可能还能列出该 run，但：

- sessions 表
- artifact 路径
- runtime session
- worktree 细节

都可能不可见。

用户需要知道这是“局部 detail unavailable”，不是整个 dashboard 崩了。

### 3. 某个 run 缺 report detail

如果 `report --run` 失败，用户最关心的字段会缺失：

- overallStatus
- recommendation
- canMerge
- mainClean

这时必须有可读解释，而不是留空或出现 `undefined`。

### 4. workflow 对 MVP run 不可用

这是 **expected degradation**，不是错误。

如果页面把它渲染成红色错误，会误导用户认为 run 有问题。

### 5. workflow 对 full run 异常失败

这和 MVP run 完全不同。

如果 full workflow run 的 `inspect --workflow --json` 返回 error，这应该被视为局部异常，而不是正常降级。

### 6. JSON parse 失败时只看到命令失败，不知道哪个子命令坏了

例如：

- `status --json` 是好的
- `status --run` 也是好的
- 但某个 `report --run` stdout 不是合法 JSON

用户应该能知道：

- 是哪个子命令坏了
- 影响了哪个 run
- dashboard 为什么没有该部分信息

### 7. report 缺字段时页面可能显示 undefined

常见危险字段：

- `report.decision`
- `report.merge`
- `report.worktree`
- `report.gates`

V2.2 必须禁止普通用户看到：

- `undefined`
- `null`
- `[object Object]`

### 8. merge / worktree 信息缺失时无法判断 blocked 原因

如果：

- `merge` 缺失
- `worktree` 缺失
- `blockedReason` 缺失
- `mainDirtyFiles` 缺失

页面不能胡编理由，也不能假装一切正常。

### 9. sessions 为空但 run 已完成时容易误解

用户会问：

- 是 run 真的完成了但没记录 session
- 还是 dashboard 数据坏了

V2.2 需要把它归类为 **inconsistent data warning**。

## V2.2 产品目标

V2.2 必须明确达到以下目标：

1. Dashboard 不因为单个 run 的局部错误整体崩溃
2. 页面不出现 `undefined`
3. 页面不出现 `null`
4. 页面不出现 `[object Object]`
5. 每个错误状态都有用户可读解释
6. MVP run 降级不是错误
7. full workflow 异常才显示 error
8. 缺字段时统一使用 `N/A` 或 `unknown`
9. 保持现有 JSON schema 不变
10. 保持现有 dashboard 数据采集路径不变

## 非目标

V2.2 不做：

1. server
2. watch
3. `dashboard --open`
4. 自动刷新
5. 执行按钮
6. artifact markdown render
7. 修改 `report/status/inspect` schema
8. 修改 dashboard 数据采集路径
9. 引入外部依赖

## 状态分类模型

V2.2 需要建立清晰的状态分类模型。所有异常都必须归入以下 6 类之一。

| 分类 | 定义 | 触发条件 | 用户看到什么 | 命令 exit 0 | 是否生成 HTML | 是否写入 dashboard model |
|---|---|---|---|---|---|---|
| `empty_state` | 没有数据可展示，但不是系统错误 | 没有 runs | Empty Dashboard + 下一步建议 | 建议 `0` | 建议 `是` | `是` |
| `global_fatal_error` | dashboard 无法构建任何可信数据 | `status --json` 失败 / 非法 JSON / output 写入失败等 | 命令失败，不生成正常 HTML | `否` | `否` | `否` |
| `per_run_recoverable_error` | 某个 run 的 detail 子命令失败，但其他 run 可用 | `status --run` / `report --run` 局部失败 | 该 run 局部 error，其他正常 | `是` | `是` | `是` |
| `expected_degradation` | 预期内的降级，不应显示为错误 | MVP run 的 `NOT_FULL_WORKFLOW` | `workflow unavailable / MVP run` | `是` | `是` | `是` |
| `missing_field_fallback` | JSON 成功，但缺字段 | `decision` / `merge` / `sessions` 等字段缺失 | `N/A` / `unknown` / `Not available` | `是` | `是` | `是` |
| `inconsistent_data_warning` | 数据结构合法，但语义可疑 | run completed 且 sessions 为空 | warning 文案，但继续展示 | `是` | `是` | `是` |

### 分类解释

#### 1. empty_state

定义：

- 系统可运行
- 只是当前没有 runs 可展示

推荐触发条件：

- `status --json` 成功
- `runs` 数组为空

推荐用户展示：

- `No runs found`
- `Run a task first: moreagent start --once --task "..."`

推荐命令行为：

- `exit 0`
- 生成 empty dashboard HTML

原因：

- “没有数据”不是系统错误
- 生成 empty dashboard 比直接失败更利于用户理解下一步

#### 2. global_fatal_error

定义：

- dashboard 连基础数据都无法建立

典型触发：

- `status --json` 执行失败
- `status --json` 返回 error JSON
- `status --json` stdout 非法 JSON
- `--run` 指定 run 不在预取范围
- output 写入失败
- data script 序列化失败

用户看到：

- 命令级错误
- 不生成正常 HTML

#### 3. per_run_recoverable_error

定义：

- 某个 run 的 detail 不可用，但 dashboard 仍可展示其他信息

典型触发：

- `status --run` error
- `report --run` error
- 某个 run detail stdout 非法 JSON

用户看到：

- 该 run 局部 `Detail unavailable` / `Report unavailable`
- 其他 run 正常

#### 4. expected_degradation

定义：

- 这是产品设计允许的降级，不算异常

典型触发：

- MVP run 的 `inspect --workflow --json` 返回 `NOT_FULL_WORKFLOW`

用户看到：

- `Workflow unavailable`
- `MVP run — workflow not available`

#### 5. missing_field_fallback

定义：

- JSON 对象存在，但某些字段缺失

典型触发：

- `report.decision` 缺失
- `report.merge` 缺失
- `report.worktree` 缺失
- `sessions` 缺失

用户看到：

- `unknown`
- `N/A`
- `Not available`

#### 6. inconsistent_data_warning

定义：

- 数据结构合法，但语义上可疑

典型触发：

- run completed 但 sessions 为空
- `hasRepair = true` 但 `repairCount` 缺失
- `blockedReason` 缺失但 recommendation 是 `BLOCKED`

用户看到：

- warning，不阻断页面

## 全局失败规则

这一节定义哪些情况 dashboard 命令必须失败。

## 1. `status --json` 失败

触发条件：

- 子进程执行失败
- 非 0 且 stdout 不是合法 error JSON

结果：

- 分类：`global_fatal_error`
- 命令失败
- 不生成正常 HTML

原因：

- `status --json` 是 dashboard 的根索引，没有它无法确定 runs 列表

## 2. `status --json` 返回 error JSON

例如：

```json
{ "error": { "code": "NO_RUNS", "message": "No runs found." } }
```

推荐决策：

- 如果 `code === NO_RUNS`：
  - **推荐不失败**
  - 生成 empty dashboard
- 其他 error：
  - 命令失败

原因：

- `NO_RUNS` 是 empty state，不是 fatal
- 其他 error 表示索引层不可用

## 3. `status --json` stdout 非合法 JSON

结果：

- `global_fatal_error`
- 命令失败
- 不生成正常 HTML

## 4. runs 为空时是失败还是生成 empty dashboard

推荐：

- **生成 empty dashboard**
- `exit 0`

原因：

1. 对用户更友好
2. 可以明确显示下一步
3. “没有 runs”不是实现错误

## 5. `--run` 指定 run 不在预取范围

触发条件：

- `status --json` 成功
- `--run` 值不在最近 `limit` 个 runs 中

结果：

- 命令失败
- 不生成正常 HTML

原因：

- 当前 dashboard 的预取模型是固定最近 N 个 runs
- 若指定 run 不在范围内，不能偷偷回退成 latest

## 6. output 写入失败

结果：

- `global_fatal_error`
- 命令失败

## 7. dashboard data script 序列化失败

例如：

- data 中出现无法安全序列化的内容

结果：

- `global_fatal_error`
- 命令失败

## 局部错误规则

这一节定义哪些情况 dashboard 应继续生成。

## 1. `status --run` 失败

处理规则：

- 继续生成 HTML
- `runDetailsById[runId].status = null`
- 写入 `statusError`
- UI 各依赖 `status` 的区域显示 fallback

## 2. `report --run` 失败

处理规则：

- 继续生成 HTML
- `runDetailsById[runId].report = null`
- 写入 `reportError`
- Summary / Gate / Merge 等区域显示 fallback

## 3. `inspect --workflow` 对 MVP run 返回 `NOT_FULL_WORKFLOW`

处理规则：

- 分类：`expected_degradation`
- 不视为错误
- `workflow = null`
- 写入 `workflowError.code = NOT_FULL_WORKFLOW`
- 页面显示 `workflow unavailable / MVP run`

## 4. `inspect --workflow` 对 full run 返回其他 error

处理规则：

- 分类：`per_run_recoverable_error`
- 不让整个 dashboard 失败
- 页面显示 workflow error

## 5. 单个 run detail JSON parse 失败

处理规则：

- 分类：`per_run_recoverable_error`
- 该 run 对应 detail 标记 error
- 其他 runs 继续

## 6. `report` 缺 `decision`

处理规则：

- 页面不显示 `undefined`
- Summary 中：
  - overallStatus = `unknown`
  - recommendation = `unknown`
- 同时显示：
  - `Missing decision data`

## 7. `report` 缺 `merge`

处理规则：

- Merge 区块显示：
  - `Merge readiness unavailable`
- 不胡编 blocked/ready 原因

## 8. `report` 缺 `worktree`

处理规则：

- worktree exists = `unknown`
- path = `N/A`
- branch = `N/A`

## 9. sessions 缺失或为空

处理规则：

- Sessions 区块显示：
  - `Sessions unavailable`
  - 或 `No session data recorded`

如果 run 已 completed，同时 sessions 为空：

- 额外显示 warning

## 页面展示规则

这一节按区域定义精确展示行为。

## A. Sidebar

### status 缺失怎么显示

规则：

- 状态 badge 显示 `unknown`
- 样式使用中性警示色

### recommendation 缺失怎么显示

规则：

- recommendation badge 显示 `unknown`

### failed / running / merge-ready marker 如何降级

规则：

- 若无可靠 recommendation / decision：
  - 不显示 fake 的 `merge ready`
- 若 `run.status` 有值：
  - 仍以 `run.status` 为主
- 若 `run.status` 也缺失：
  - 使用 `unknown`

### task 缺失怎么显示

规则：

- 显示 `Untitled task`

## B. Enhanced Summary

### decision 缺失怎么显示

规则：

- Overall Status = `unknown`
- Recommendation = `unknown`
- 错误说明：`Missing decision data`

### canMerge / mainClean / worktree.exists 缺失怎么显示

规则：

- 布尔缺失时不显示 true/false
- 显示 `N/A`

### status / report 局部错误如何显示

规则：

- 若 `reportError` 存在：
  - 显示 `Report unavailable`
- 若 `statusError` 存在：
  - 显示 `Detail unavailable`

## C. Workflow Progress

### MVP run expected degradation

显示：

- `Workflow unavailable`
- `MVP run — workflow not available`

### full run workflow error

显示：

- `Workflow unavailable`
- `Workflow data could not be loaded for this full workflow run`

### completedPhases 缺失

规则：

- full run：当作空数组
- 不显示 `undefined`

### failedPhase 缺失

规则：

- 显示 `None`

### totalPhases 缺失

规则：

- full workflow 用固定值 `9`
- 不依赖输入字段

## D. Gate / Test / Review

### unknown 值

规则：

- 文案用 `unknown`
- 样式中性警示色

### 缺字段 fallback

规则：

- 缺 `gates`：四项都显示 `unknown`

### reportError 时显示什么

规则：

- 整个区块显示：
  - `Report unavailable`

不拼接假 gate 结果。

## E. Merge Readiness

### merge 缺失

显示：

- `Merge readiness unavailable`

### worktree 缺失

显示：

- `Worktree information unavailable`

### blockedReason 缺失

规则：

- 不胡编
- 显示：
  - `Blocked reason not available`

### mainDirtyFiles 缺失

规则：

- 若 `mainClean === false` 但 `mainDirtyFiles` 缺失：
  - 只显示 `Main repository is not clean`
  - 不伪造文件列表

### explanation 如何不胡编

规则：

- 只使用现有字段：
  - `recommendation`
  - `canMerge`
  - `mainClean`
  - `worktree.exists`
  - `blockedReason`
- 如果不足以解释，显示：
  - `Not available`

## F. Repair Sessions

### hasRepair false

显示：

- `No repair sessions`

### repairCount 缺失

显示：

- `N/A`

### repairSessions 缺失

显示：

- `Repair session details unavailable`

### sessions 中 repair 命名异常

规则：

- 不试图强行解释轮次
- 显示原 sessionName
- 同时标记：
  - `Unrecognized repair session name`

## G. Sessions Table

### sessions 为空

显示：

- `No session data recorded`

### startedAt / completedAt 缺失

显示：

- `N/A`

### durationSeconds 缺失

显示：

- `N/A`

### artifactDir 缺失

显示：

- `Not available`

### runtimeSessionId 缺失

显示：

- `Not available`

## H. JSON / Debug

### statusError / reportError / workflowError 展示格式

建议格式：

- Title
- error code
- error message

例如：

```text
Report unavailable
Code: RUN_NOT_FOUND
Message: Run not found: run-xxx
```

### 原始 JSON 为 null 时显示什么

规则：

- 显示：
  - `Not available`

### error code / message 如何显示

规则：

- code 单独一行
- message 单独一行
- 不显示原始 `[object Object]`

## 文案规范

这一节定义固定文案，避免实现时自由发挥。

## 必须提供的具体文案

- `No runs found`
- `Detail unavailable`
- `Report unavailable`
- `Workflow unavailable`
- `MVP run — workflow not available`
- `Missing decision data`
- `Merge readiness unavailable`
- `Sessions unavailable`

## `Unknown / N/A / Not available` 的区别

### `unknown`

含义：

- 字段应该存在一个判定值
- 但当前没有可靠值

适用：

- overallStatus
- recommendation
- gate/test/review

### `N/A`

含义：

- 该字段对当前上下文不适用，或不是布尔真假问题

适用：

- duration
- canMerge 缺失时展示位
- worktree exists 缺失时展示位

### `Not available`

含义：

- 该数据本应可显示，但当前没有拿到

适用：

- artifactDir
- runtimeSessionId
- worktree path
- raw JSON panel

## 验收标准

以下验收标准必须可测试：

1. 没有 runs 时，Dashboard 生成 empty state HTML，而不是直接崩溃
2. empty state 页面包含 `No runs found`
3. empty state 页面包含下一步建议
4. `status --run` 失败时，dashboard 仍生成
5. `status --run` 失败时，页面出现 `Detail unavailable`
6. `report --run` 失败时，dashboard 仍生成
7. `report --run` 失败时，页面出现 `Report unavailable`
8. MVP run 的 `NOT_FULL_WORKFLOW` 不被渲染成错误
9. MVP run 页面显示 `MVP run — workflow not available`
10. full workflow 的 workflow error 显示为 error，而不是 MVP 降级
11. 缺 `decision` 时，页面显示 `Missing decision data`
12. 缺 `merge` 时，页面显示 `Merge readiness unavailable`
13. sessions 为空时，页面显示 `No session data recorded` 或 `Sessions unavailable`
14. 页面主 UI 中不出现 `undefined`
15. 页面主 UI 中不出现字面量 `null`
16. 页面主 UI 中不出现 `[object Object]`
17. `blockedReason` 缺失时不胡编 blocked explanation
18. `mainDirtyFiles` 缺失时不伪造文件列表
19. JSON / Debug 区仍可查看 error code 与 message
20. 保持现有 JSON schema 不变
