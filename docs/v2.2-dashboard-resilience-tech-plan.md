# V2.2 Dashboard 空状态 / 错误态 / 数据稳定性补强 Technical Plan

## 目标

V2.2 的目标不是新增 Dashboard 功能，而是让 Dashboard 在以下场景下稳定工作：

1. 空数据
2. 局部错误
3. 子命令 error JSON
4. 子命令 stdout 非法 JSON
5. 缺字段
6. 语义不一致数据

实现原则：

- 保持现有 JSON schema 不变
- 保持现有 dashboard 数据采集路径不变
- 所有 UI 在渲染前先进入 Safe ViewModel
- 不在模板中直接深层访问脆弱字段

## 当前实现分析

## 1. `dashboard.ts` 当前数据流

当前主要链路：

1. `dashboardCommand(options)`
2. `buildDashboardModel(selectedRun, limit)`
3. `callMoreagentJson(args)`
4. `renderDashboardHtml(model)`
5. `writeDashboardHtml(outputPath, html)`

## 2. `buildDashboardModel()`

当前职责：

1. 调 `status --json`
2. 取最近 N 个 runs
3. 选定 `selectedRunId`
4. 对每个 run 调：
   - `status --run`
   - `report --run`
   - `inspect --workflow`
5. 填入：
   - `status`
   - `report`
   - `workflow`
   - `statusError`
   - `reportError`
   - `workflowError`

当前优点：

- 已支持 per-run 局部错误
- 已支持 MVP run 的 workflow 降级
- 已支持 `runDetailsById`

当前不足：

1. 仍偏 raw data model，未做统一 Safe ViewModel 归一化
2. 模板层容易深层访问可缺失字段
3. 某些 fallback 可能分散在 UI 代码中
4. “预期降级”和“异常错误”目前边界还不够清晰

## 3. `callMoreagentJson()`

当前职责：

1. 调 CLI 子命令
2. 读取 stdout
3. `JSON.parse`
4. 若返回 `{ error }` 则映射为 error

当前优点：

- 能统一捕获 error JSON

当前风险点：

1. parse 失败时只有一段泛化 message
2. 不一定能明确指出是哪个子命令坏了
3. 对错误类型的分类还不够结构化

## 4. `renderDashboardHtml()`

当前职责：

- 直接把 dashboard model 嵌入 HTML
- 内联 CSS / JS
- 页面内完成渲染

当前风险点：

1. 直接从 raw model 渲染，容易出现缺字段访问
2. 缺字段时容易渲染出 `undefined`
3. 局部错误解释可能不一致

## 5. 当前错误处理方式

当前大致是：

- 全局索引失败：命令失败
- per-run detail 失败：记录 `statusError/reportError/workflowError`
- workflow 对 MVP run：降级

这已经是 V2.0/V2.1 的良好基础。

V2.2 要做的是进一步细化：

1. error 分类
2. fallback 规则
3. safe normalization
4. UI 文案标准化

## 6. 当前风险点总结

1. raw model 直接渲染
2. 缺字段 fallback 不统一
3. 错误文案可能分散
4. “empty / fatal / recoverable / degradation / warning” 还未系统化

## 建议新增 helper

V2.2 建议新增以下 helper。所有 helper 都在 `dashboard.ts` 内部完成，不 export 到其他命令。

## 1. `safeText(value, fallback)`

### 输入

```ts
safeText(value: unknown, fallback: string): string
```

### 输出

- 若 `value` 是非空字符串，返回 trim 后字符串
- 若 `value` 是数字 / 布尔，转字符串
- 若 `value` 为 `null/undefined/''`，返回 `fallback`
- 若 `value` 是对象，返回 `fallback`

### 目的

防止 UI 出现：

- `undefined`
- `null`
- `[object Object]`

## 2. `safeBool(value)`

### 输入

```ts
safeBool(value: unknown): true | false | null
```

### 输出

- `true`
- `false`
- `null` 表示未知

### 目的

用于：

- `canMerge`
- `canResume`
- `mainClean`
- `worktree.exists`

## 3. `getNested(obj, path, fallback)`

### 输入

```ts
getNested<T>(obj: unknown, path: string[], fallback: T): T
```

### 输出

- 若路径存在，返回路径值
- 否则返回 `fallback`

### 目的

统一安全访问深层字段，例如：

- `report.report.decision.recommendation`
- `report.report.merge.mainClean`

## 4. `normalizeDecision(report)`

### 输入

```ts
normalizeDecision(report: any | null): {
  overallStatus: string;
  recommendation: string;
  canResume: true | false | null;
  isMissing: boolean;
  warning?: string;
}
```

### 输出规则

- 缺 decision：`overallStatus = 'unknown'`, `recommendation = 'unknown'`, `isMissing = true`
- 有值时使用真实值

## 5. `normalizeMerge(report)`

### 输入

```ts
normalizeMerge(report: any | null): {
  canMerge: true | false | null;
  mainClean: true | false | null;
  blockedReason: string | null;
  mainDirtyFiles: string[];
  isMissing: boolean;
}
```

### 输出规则

- 缺 `merge`：`isMissing = true`
- 缺 `mainDirtyFiles`：返回空数组，但不伪造 dirty file

## 6. `normalizeWorkflow(details)`

### 输入

```ts
normalizeWorkflow(details: RunDetailEntry): WorkflowViewModel
```

### 输出规则

- `workflowError.code === NOT_FULL_WORKFLOW`：
  - `mode = 'mvp_degradation'`
- full run workflow 正常：
  - `mode = 'full_ok'`
- full run workflow error：
  - `mode = 'full_error'`

## 7. `normalizeSessions(statusRun)`

### 输入

```ts
normalizeSessions(statusRun: any | null): SessionsViewModel
```

### 输出规则

- 缺 sessions：`state = 'unavailable'`
- 空数组：`state = 'empty'`
- 正常数组：`state = 'ok'`

## 8. `classifyRunError(details)`

### 输入

```ts
classifyRunError(details: RunDetailEntry): {
  kind: 'none' | 'status_error' | 'report_error' | 'workflow_error' | 'warning';
  messages: string[];
}
```

### 目的

让一个 run 的局部问题有统一分类结果，而不是分散在模板判断。

## 9. `renderErrorBox(title, error)`

### 输入

```ts
renderErrorBox(title: string, error: { code?: string; message?: string } | null): string
```

### 输出

- 标准化 HTML 片段
- 内含：
  - 标题
  - code
  - message

## 10. `renderEmptyState(type)`

### 输入

```ts
renderEmptyState(type: 'no_runs' | 'no_sessions' | 'no_repair' | 'detail_unavailable'): string
```

### 输出

- 标准化 empty state HTML

## 11. `buildSafeDashboardViewModel(run, details)`

### 输入

```ts
buildSafeDashboardViewModel(run: any, details: RunDetailEntry): SafeRunViewModel
```

### 输出

- 一个已归一化、可直接给模板使用的安全视图模型

## 错误分类伪代码

## 1. `classifyDashboardGlobalError()`

```ts
function classifyDashboardGlobalError(statusListResult, selectedRun, outputPath) {
  if (statusListResult.execFailed) {
    return { kind: 'global_fatal_error', reason: 'status_json_exec_failed' };
  }

  if (statusListResult.parseFailed) {
    return { kind: 'global_fatal_error', reason: 'status_json_parse_failed' };
  }

  if (statusListResult.errorCode === 'NO_RUNS') {
    return { kind: 'empty_state', reason: 'no_runs' };
  }

  if (statusListResult.errorCode) {
    return { kind: 'global_fatal_error', reason: 'status_json_error' };
  }

  const prefetchedRuns = Array.isArray(statusListResult.runs)
    ? statusListResult.runs
    : [];

  const matched = selectedRun
    ? prefetchedRuns.find((r) => r.id === selectedRun || r.id.startsWith(selectedRun))
    : null;

  if (selectedRun && !matched) {
    return { kind: 'global_fatal_error', reason: 'selected_run_out_of_range' };
  }

  if (!outputPath) {
    return { kind: 'global_fatal_error', reason: 'output_path_invalid' };
  }

  return { kind: 'ok' };
}
```

说明：

- `--run` 只在 **prefetched range** 内匹配
- 匹配规则应与当前 dashboard 行为一致：
  - 先取最近 `limit` 个 runs
  - 再在这批 run 中用 `id === selectedRun || id.startsWith(selectedRun)` 匹配
- 不能在全量 runs 外再做额外查询，否则会破坏 V2.0 的静态预取模型

## 2. `classifyRunDetailError()`

```ts
function classifyRunDetailError(runSummary, details) {
  const isFull = runSummary.profile === 'full';

  if (details.statusError) {
    return { kind: 'per_run_recoverable_error', source: 'status' };
  }

  if (details.reportError) {
    return { kind: 'per_run_recoverable_error', source: 'report' };
  }

  if (details.workflowError?.code === 'NOT_FULL_WORKFLOW' && !isFull) {
    return { kind: 'expected_degradation', source: 'workflow' };
  }

  if (details.workflowError && isFull) {
    return { kind: 'per_run_recoverable_error', source: 'workflow' };
  }

  if (details.status && Array.isArray(details.status.run?.sessions) && details.status.run.sessions.length === 0 && details.status.run?.status === 'completed') {
    return { kind: 'inconsistent_data_warning', source: 'sessions' };
  }

  return { kind: 'none' };
}
```

## 3. `normalizeReport()`

```ts
function normalizeReport(report) {
  const reportNode = getNested(report, ['report'], null);
  const decisionNode = getNested(reportNode, ['decision'], null);
  const mergeNode = getNested(reportNode, ['merge'], null);
  const worktreeNode = getNested(reportNode, ['worktree'], null);
  const gatesNode = getNested(reportNode, ['gates'], null);

  return {
    decision: decisionNode
      ? {
          overallStatus: safeText(decisionNode.overallStatus, 'unknown'),
          recommendation: safeText(decisionNode.recommendation, 'unknown'),
          canResume: safeBool(decisionNode.canResume),
        }
      : {
          overallStatus: 'unknown',
          recommendation: 'unknown',
          canResume: null,
          missing: true,
        },
    merge: mergeNode
      ? {
          canMerge: safeBool(mergeNode.canMerge),
          mainClean: safeBool(mergeNode.mainClean),
          blockedReason: safeText(mergeNode.blockedReason, ''),
          mainDirtyFiles: Array.isArray(mergeNode.mainDirtyFiles) ? mergeNode.mainDirtyFiles.filter(Boolean) : [],
        }
      : { missing: true },
    worktree: worktreeNode
      ? {
          exists: safeBool(worktreeNode.exists),
          path: safeText(worktreeNode.path, 'Not available'),
          branch: safeText(worktreeNode.branch, 'Not available'),
        }
      : { missing: true },
    gates: {
      prdGate: safeText(gatesNode?.prdGate, 'unknown'),
      techGate: safeText(gatesNode?.techGate, 'unknown'),
      test: safeText(gatesNode?.test, 'unknown'),
      review: safeText(gatesNode?.review, 'unknown'),
    },
  };
}
```

## 4. `normalizeWorkflow()`

```ts
function normalizeWorkflow(runSummary, details) {
  const isFull = safeText(runSummary.profile, 'mvp') === 'full';

  if (details.workflowError?.code === 'NOT_FULL_WORKFLOW' && !isFull) {
    return {
      mode: 'mvp_degradation',
      title: 'Workflow unavailable',
      subtitle: 'MVP run — workflow not available',
      totalPhases: 0,
      completedPhases: [],
      failedPhase: null,
      phaseItems: [],
    };
  }

  if (details.workflowError) {
    return {
      mode: 'full_error',
      title: 'Workflow unavailable',
      subtitle: 'Workflow data could not be loaded for this full workflow run',
      totalPhases: 9,
      completedPhases: [],
      failedPhase: null,
      phaseItems: [],
      errorCode: safeText(details.workflowError.code, 'INTERNAL_ERROR'),
      errorMessage: safeText(details.workflowError.message, 'Workflow unavailable'),
    };
  }

  const workflow = getNested(details.workflow, ['workflow'], null);
  const completedPhases = Array.isArray(workflow?.completedPhases) ? workflow.completedPhases : [];
  const failedPhase = workflow?.failedPhase ? String(workflow.failedPhase) : null;
  const allPhases = ['brain','prd','prd-review','prd-gate','tech-plan','tech-gate','implementation','test','review'];

  return {
    mode: 'full_ok',
    title: 'Workflow Progress',
    subtitle: null,
    totalPhases: 9,
    completedPhases,
    failedPhase,
    phaseItems: allPhases.map((name) => ({
      name,
      state: failedPhase === name ? 'failed' : completedPhases.includes(name) ? 'completed' : 'pending',
    })),
  };
}
```

## 5. `buildMergeExplanationSafe()`

```ts
function buildMergeExplanationSafe(merge, worktree, decision) {
  if (merge.missing) {
    return {
      title: 'Merge readiness unavailable',
      items: ['Report did not include merge readiness data.'],
    };
  }

  if (decision.recommendation === 'MERGE_READY') {
    return {
      title: 'Ready reason',
      items: [
        'Run passed all required checks.',
        merge.canMerge === true ? 'Run is mergeable.' : 'Mergeability not available.',
        merge.mainClean === true ? 'Main repository is clean.' : 'Main repository cleanliness is unknown.',
        worktree.exists === true ? 'Worktree exists.' : 'Worktree status is unknown.',
      ],
    };
  }

  if (decision.recommendation === 'BLOCKED') {
    const items = [];
    if (merge.canMerge === false) items.push('Run is not mergeable.');
    if (merge.mainClean === false) items.push('Main repository has uncommitted changes.');
    if (worktree.exists === false) items.push('Worktree is missing.');
    if (items.length === 0) items.push(safeText(merge.blockedReason, 'Blocked reason not available.'));
    return {
      title: 'Blocked reason',
      items,
    };
  }

  return {
    title: 'Not available',
    items: ['Merge explanation is not available for this run state.'],
  };
}
```

## 渲染策略

V2.2 必须明确渲染顺序：

1. raw dashboard model 继续嵌入 JSON
2. 页面启动后先读取 raw model
3. 在浏览器端 JS 渲染前，对选中 run 调 `buildSafeDashboardViewModel()`
4. 页面渲染只使用 Safe ViewModel

绝对禁止：

- 在模板里直接深层访问：
  - `details.report.report.decision.overallStatus`
  - `details.status.run.sessions`
  - `details.workflow.workflow.completedPhases`

原因：

- 这些路径都可能缺失

推荐模式：

```ts
const safe = buildSafeDashboardViewModel(run, details);
renderSummary(safe.summary);
renderWorkflow(safe.workflow);
renderMerge(safe.merge);
renderSessions(safe.sessions);
```

## Safe ViewModel 的运行位置

V2.2 必须明确 Safe ViewModel 的构建位置，避免实现者做出两套 normalization。

推荐方案：

1. `DashboardModel` 仍然是 raw model
2. raw model 继续原样嵌入 `window.__MOREAGENT_DASHBOARD_DATA__`
3. Safe ViewModel 在浏览器端 JS 渲染前构建
4. UI 只消费 Safe ViewModel

原因：

1. 这样不改变 V2.0 的嵌入协议
2. raw model 仍可供 JSON / Debug 区直接展示
3. 归一化逻辑集中在一处，不会出现“生成阶段 normalize 一次，浏览器端再 normalize 一次”的双重逻辑

如果实现者决定把部分 helper 放在 TypeScript 生成阶段，必须满足：

1. 仅做纯数据安全处理
2. 不生成第二套独立 schema
3. 不与浏览器端再重复 normalize

V2.2 推荐仍以**浏览器端单次 normalize**为主，不建议双阶段 normalize。

## 测试可实现性说明

V2.2 的测试要分三类管理，否则很容易在文档里写出“理论上可测、实际无法稳定落地”的用例。

## A. 当前 CLI E2E 可直接测试

这些测试当前就能通过 CLI + fixture 稳定落地：

1. no runs
2. workflow `NOT_FULL_WORKFLOW`
3. empty sessions
4. `dashboard --run / --limit / --output` 行为不回退
5. JSON / Debug 区仍可访问

标记：

- `E2E feasible now`

## B. 需要通过缺字段 fixture 间接测试

这些测试通常不需要 test hook，但要通过人为构造不完整 JSON / run detail fixture 触发：

1. missing task
2. missing sessions
3. missing workflow fields

标记：

- `E2E feasible now (fixture-driven)`

## C. 需要新增 test hook 或调整实现才能稳定测试

这些场景如果要做精确分支验证，建议通过 helper 测试或内部 test hook 解决：

1. `status --run` 局部 error
2. `report --run` 局部 error
3. missing decision
4. missing merge
5. missing worktree

标记：

- `requires helper test`

若本轮不引入 test hook，则这些测试只能做“间接字符串级覆盖”，精度会下降。

## 是否允许新增 test-only hook

V2.2 需要在“测试精度”和“CLI 污染”之间做明确取舍。

推荐方案：

1. 不新增外部依赖
2. 不新增 CLI 参数
3. 可以在 `dashboard.ts` 内保留纯函数 normalizer
4. **允许导出一个内部测试对象**，例如：

```ts
export const __dashboardTestHooks = {
  safeText,
  safeBool,
  getNested,
  normalizeDecision,
  normalizeMerge,
  normalizeWorkflow,
  normalizeSessions,
  classifyRunError,
  buildSafeDashboardViewModel,
};
```

### 该对象是否属于 public API

明确结论：

- **不属于 public API**
- 仅用于测试
- 命名上必须以 `__` 前缀明确表达内部性质

### 如何避免污染 CLI 使用

规则：

1. 不在 help 中暴露
2. 不新增 CLI 参数
3. 不新增对外文档承诺
4. 仅在测试代码中 import 使用

### 为什么推荐这一路径

原因：

1. V2.2 的大量 resilience 分支属于 normalize 层逻辑
2. 这些逻辑若只依赖黑盒 E2E，很难精确覆盖 missing decision / missing merge / missing worktree 等缺字段分支
3. 暴露内部 test hooks 比新增 CLI 参数更干净

### 替代方案

替代方案是：

1. 不导出 helper
2. 全部通过字符串 / fixture 级间接测试

缺点：

1. 分支定位不精准
2. 当 fallback 文案变化时，测试会脆弱
3. 某些缺字段路径不一定能稳定构造

结论：

## 风险与回滚

## 风险点

1. helper 增多后代码组织变复杂
2. 若一次性重写模板，测试波动会大
3. fallback 太多时，可能把真实错误掩盖掉

## 如何避免大改模板

建议：

1. 保留现有主要 DOM 区块
2. 优先插入 Safe ViewModel 层
3. 局部替换字段读取，不整体推翻模板

## 如何保持测试稳定

建议：

1. 新增 resilience fixture
2. 保留现有 V2.1 文案关键字
3. 不轻易重命名区块标题

## 如何回滚到 V2.1

回滚策略：

1. 保留 raw dashboard model 逻辑不变
2. 若 V2.2 Safe ViewModel 层引发问题，可回退到旧渲染 helper
3. 按 commit 级回滚 `dashboard.ts` 与新增测试

## 是否建议进入实现

建议进入实现。

原因：

1. V2.2 解决的是稳定性短板，而不是概念性新功能。
2. 当前 dashboard 已经有良好基础，resilience 增强可局部演进。
3. Safe ViewModel 方案可以显著降低 `undefined/null/[object Object]` 风险。
4. 测试边界清晰，容易验证是否真正补强。
