# V2.0 Dashboard / Web View Technical Plan

## 目标

基于 V1.8 JSON 输出和 V1.9 `report` 命令，为 `moreagent` 设计 V2.0 Dashboard / Web View 的技术方案。

只做设计，不实现。

## 约束

1. 不改现有 `src` 实现
2. 不新增依赖
3. 不引入远程服务
4. 不做登录
5. 不做自动刷新
6. 不做复杂图表

## 当前可复用能力

当前已经有三类结构化数据源：

1. `moreagent status --json`
2. `moreagent status --run <id> --json`
3. `moreagent inspect --run <id> --workflow --json`
4. `moreagent report --run <id> --json`

其中职责建议明确划分：

- `status --json`：run 列表索引
- `status --run <id> --json`：session 级详情
- `inspect --workflow --json`：workflow / gate / repair 结构
- `report --json`：总体判断、merge readiness、recommendation

## 技术决策

## 决策 1：MVP 先做静态 HTML，不先做本地 server

推荐方案：**CLI 生成 HTML 文件**

理由：

1. 与“无依赖”约束最一致
2. 测试最简单
3. 不需要长驻进程
4. 不引入端口冲突
5. 适合本地打开和归档

不选本地 dev server 作为 V2.0 MVP 的原因：

1. server 生命周期要管理
2. 需要静态资源路由
3. smoke test 会更重
4. 如果没有自动刷新，server 的价值不大

结论：

- V2.0：静态 HTML
- V2.1 以后：如果需要再加 `--serve`

## 决策 2：新增 `moreagent dashboard` 命令，比扩展 `report --html` 更清晰

两种候选：

### 方案 A：`moreagent report --html`

优点：

- 复用已有 report 概念

缺点：

- `report` 本质是单 run 报告
- Dashboard 还需要 runs 列表和 JSON/debug view
- 语义会越来越混杂

### 方案 B：`moreagent dashboard`

优点：

- 命令语义清晰
- 天然容纳 run 列表 + 详情页
- 后续加 `--serve` 也自然

缺点：

- 新增一个命令入口

结论：**选方案 B**

## 决策 3：Dashboard 数据组装层不直接读 sessions.json

MVP 技术边界建议：

- Dashboard 视图层只消费 CLI JSON 输出
- 不直接读取 `.moreagent/sessions.json`
- 不直接扫 artifact 文件

原因：

1. 避免再造一套业务解析逻辑
2. 保证 Dashboard 与 CLI 判断一致
3. 未来若 JSON schema 稳定，Web View 和 CLI 可共享协议

唯一例外：

- HTML 模板静态资源本身可由命令直接写文件

## 建议命令接口

MVP 建议：

```bash
moreagent dashboard
moreagent dashboard --latest
moreagent dashboard --run <id>
moreagent dashboard --output .moreagent/dashboard/index.html
```

建议语义：

- `moreagent dashboard`
  - 读取 `status --json`
  - 默认选中 latest run
  - 生成 HTML

- `moreagent dashboard --run <id>`
  - 聚焦单个 run
  - 仍可在页面侧栏展示最近 runs

- `--output`
  - 指定 HTML 输出路径
  - 默认可落在 `.moreagent/dashboard/`

MVP 不建议先做：

```bash
moreagent dashboard --serve
moreagent dashboard --watch
```

## 页面渲染模型

建议 HTML 为单文件：

- 一个 HTML 文件
- 内嵌 CSS
- 内嵌少量原生 JS
- 内嵌一份 `window.__MOREAGENT_DASHBOARD_DATA__`

这样不需要：

1. bundler
2. npm frontend toolchain
3. 外部静态资源

## 数据模型

建议 Dashboard 内部使用统一 ViewModel：

```ts
interface DashboardModel {
  generatedAt: string;
  runs: RunListItem[];
  selectedRunId: string | null;
  selectedRun: {
    status: StatusRunDetail | null;
    report: WorkflowReport | null;
    workflow: WorkflowInspect | null;
  };
}
```

说明：

- `runs` 来自 `status --json`
- `selectedRun.status` 来自 `status --run <id> --json`
- `selectedRun.report` 来自 `report --run <id> --json`
- `selectedRun.workflow` 来自 `inspect --run <id> --workflow --json`

页面只消费这一份聚合模型。

## 数据获取流程

## 首页/默认 latest

1. 调用 `status --json`
2. 取第一条 run 作为默认 selected run
3. 对 selected run 继续调用：
   - `status --run <id> --json`
   - `report --run <id> --json`
   - `inspect --run <id> --workflow --json`
4. 组装成 `DashboardModel`
5. 渲染 HTML

## 指定 run

1. 调用 `status --json`
2. 校验 `--run` 是否存在
3. 拉取该 run 三类详情 JSON
4. 生成 HTML

## 错误处理

Dashboard 命令应复用现有 JSON 错误风格：

- 如果 `status --json` 返回 error
- 或指定 run 不存在
- 或 workflow JSON 返回 `NOT_FULL_WORKFLOW`

MVP 处理建议：

1. 命令层退出非 0
2. 不生成成功 HTML
3. 输出文本错误

对于非 full workflow run：

- Dashboard 仍应支持显示
- 但 workflow 区块显示：
  - `profile: mvp`
  - `workflow report unavailable`

这意味着实现层不能把 `inspect --workflow --json` 失败视作整体失败，应该允许降级展示。

这是一个关键边界。

## 视图模块拆分建议

虽然 V2.0 不实现，但实现时建议拆成纯字符串渲染模块：

1. `buildDashboardModel()`
2. `renderDashboardHtml(model)`
3. `renderRunList(model)`
4. `renderRunHeader(model)`
5. `renderWorkflowSection(model)`
6. `renderGateSection(model)`
7. `renderRepairSection(model)`
8. `renderMergeSection(model)`
9. `renderSessionsSection(model)`
10. `renderJsonTabs(model)`

原则：

- 数据组装和 HTML 渲染分离
- 不在模板里现场做复杂业务判断

## 推荐 UI 结构

## 左侧栏

- 最近 runs
- 简单筛选按钮
- 当前选中 run 高亮

## 主内容

按从上到下顺序：

1. Run Header
2. Decision / Recommendation
3. Workflow Progress
4. Gate / Test / Review cards
5. Repair Sessions
6. Merge Readiness
7. Sessions Table
8. JSON / Debug tabs

## 为什么不建议 V2.0 直接做 SPA

即使不用依赖，手写 SPA 也会引入：

1. 路由状态
2. 客户端数据管理
3. 事件同步
4. 浏览器端数据加载

这些复杂度对当前目标没有价值。V2.0 更像一个“离线生成的验收页面”，不是应用平台。

## 测试方案

MVP 测试聚焦四件事：

## 1. JSON fixture 测试

准备 fixture：

1. completed + merge ready
2. completed + blocked
3. failed + repair
4. running
5. mvp run without workflow

目标：

- 验证 `DashboardModel` 组装稳定
- 验证缺失 workflow 时可降级

## 2. dashboard command smoke test

建议验收：

```bash
moreagent dashboard
moreagent dashboard --run <id>
```

检查：

- 命令 exit 0
- HTML 文件存在

## 3. HTML 输出存在与结构检查

检查输出 HTML 包含关键标识：

- `Run List`
- `Workflow Report`
- `Gate`
- `Repair Sessions`
- `Merge Readiness`
- `JSON / Debug`

无需浏览器级复杂测试，MVP 只做字符串存在检查即可。

## 4. 报告数据正确渲染

至少覆盖：

1. `MERGE_READY`
2. `BLOCKED`
3. `NEEDS_REPAIR`
4. `RUNNING`
5. `PARTIAL`

确保这些 recommendation 能在 HTML 中被准确展示。

## 风险与边界

## 风险 1：JSON schema 漂移

Dashboard 强依赖 V1.8/V1.9 schema 稳定。

建议：

- V2.0 实现前，先把 JSON 字段冻结
- 新字段可以加，旧字段不要改名

## 风险 2：非 full workflow run 的 workflow 视图不完整

MVP run 没有完整 phase 数据。

建议：

- 页面明确显示 `profile: mvp`
- workflow 区块降级，不报错

## 风险 3：单文件 HTML 体积增长

当前数据量很小，不构成问题。

V2.0 可以接受。

## 风险 4：命令内再调用 CLI 命令的实现路径

如果未来实现时用 Node 进程再调用自身 CLI，需要处理：

1. 路径定位
2. JSON parse 失败
3. 子进程 exit code

这是实现细节，不影响当前方案成立。

## 分阶段建议

## V2.0 MVP

1. `moreagent dashboard`
2. 生成静态 HTML
3. 最近 runs + 单 run 详情
4. JSON/debug tabs

## V2.1

1. `dashboard --serve`
2. 刷新按钮
3. artifact markdown 渲染

## V2.2

1. 多项目视图
2. diff 内嵌
3. merge / resume 操作入口

## 是否建议进入实现

**建议进入实现。**

原因：

1. 数据层已经存在，不需要先补后端协议。
2. MVP 可以严格收敛为静态 HTML 生成，复杂度可控。
3. 不需要引入依赖即可落地。
4. 测试边界清晰，适合快速交付一个可验收版本。

## 实现优先级建议

1. 先实现 `dashboard` 命令的数据组装
2. 再实现单文件 HTML 渲染
3. 最后补 smoke test 和 fixture

不要反过来先做 server。
