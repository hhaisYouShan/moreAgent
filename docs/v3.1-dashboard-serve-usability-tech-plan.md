# V3.1 Dashboard Serve / Watch Usability Technical Plan

## 目标

V3.1 的目标不是增加新的服务模式能力，而是在已完成的 V3.0 `dashboard --serve / --watch / --open` 基础上增强：

1. serve startup output
2. 页面刷新状态可见性
3. `/data.json` 错误可解释性
4. 使用文档完整性

实现边界：

1. 不引入 React/Vue
2. 不引入 Express
3. 不引入 WebSocket/SSE
4. 不增加执行型按钮
5. 不重做 DashboardModel 主 schema，除非采用兼容附加层

## 当前基线假设

假设 V3.0 已经具备：

1. `dashboard --serve`
2. `dashboard --serve --watch`
3. `dashboard --serve --open`
4. `GET /`
5. `GET /data.json`
6. `GET /health`
7. selected run persist
8. `Refresh data`

V3.1 在这个基线之上做增强，不回头重做 V3.0 主结构。

## 核心技术决策

## 1. 不修改 DashboardModel 主 schema

V3.1 推荐不去改变 `/data.json` 的核心 DashboardModel 结构。

原因：

1. V3.0 已建立 status/report/inspect 聚合模型
2. 大改 schema 会增加回归风险
3. V3.1 重点是 runtime usability，不是数据模型扩容

### 兼容策略

新增一层 runtime-only metadata，而不是重写 DashboardModel。

推荐：

1. 保持 `window.__MOREAGENT_DASHBOARD_DATA__` 为现有 dashboard model
2. 增加 `window.__MOREAGENT_DASHBOARD_RUNTIME__`

例如：

```ts
{
  serveMode: true,
  watchEnabled: boolean,
  refreshIntervalMs: number | null,
  startup: {
    host: string,
    port: number,
    url: string,
    selectedRun: string | null,
    limit: number
  }
}
```

这样：

1. 不破坏现有数据消费逻辑
2. serve/watch runtime 状态有明确归宿
3. 静态模式可以不给或给最小 runtime

## 2. 页面刷新状态是前端 runtime state，不写回 DashboardModel

刷新状态如：

1. `lastRefreshedAt`
2. `isRefreshing`
3. `lastRefreshError`
4. `lastRefreshStatus`
5. `isShowingStaleData`

推荐仅存在浏览器端 JS 内存中。

原因：

1. 它们不是 run/domain data
2. 它们是页面运行态
3. 不应污染 `status/report/inspect` 的产物模型

## 3. partial error 与 hard error 要分层处理

V3.1 需要区分：

### Hard refresh error

例如：

1. HTTP 500
2. JSON parse failed
3. selected run truly missing

结果：

1. 本次刷新失败
2. 保留旧数据
3. 标记 `isShowingStaleData = true`

### Partial refresh success

例如：

1. `/data.json` 整体可解析
2. 但某个 run 的 `statusError`
3. 或 `reportError`
4. 或 `workflowError`

结果：

1. 整体刷新成功
2. 页面更新数据
3. runtime status 显示 `refreshed with partial errors`

## 前端 runtime 状态设计

## 新增 runtime state 字段

浏览器端建议维护：

```ts
type RefreshState =
  | { kind: 'idle'; lastRefreshedAt: string | null; isShowingStaleData: false }
  | { kind: 'refreshing'; lastRefreshedAt: string | null; trigger: 'manual' | 'watch' }
  | { kind: 'error'; lastRefreshedAt: string | null; errorType: string; message: string; isShowingStaleData: true }
  | { kind: 'partial'; lastRefreshedAt: string; partialErrorCount: number; isShowingStaleData: false };
```

这不需要真的写 TypeScript 类型到浏览器端，只是设计约束。

## 手动刷新实现建议

## 新增 helper

### `fetchDashboardData()`

职责：

1. fetch `/data.json`
2. 解析 HTTP 状态
3. 解析 JSON
4. 返回统一结果对象

推荐返回：

```ts
type FetchResult =
  | { ok: true; data: DashboardModel; partialErrors: DashboardErrorSummary[] }
  | { ok: false; errorType: 'http' | 'parse' | 'run_not_found' | 'unknown'; message: string };
```

### `refreshDashboard(trigger)`

职责：

1. 更新 runtime state 为 `refreshing`
2. 调用 `fetchDashboardData()`
3. 成功时更新 `window.__MOREAGENT_DASHBOARD_DATA__`
4. 调用现有 render 流程重绘
5. 失败时保留旧数据并显示 stale state

输入：

1. `trigger = 'manual' | 'watch'`

### `detectPartialErrors(data)`

职责：

1. 遍历 `runDetailsById`
2. 统计：
   - `statusError`
   - `reportError`
   - `workflowError`
3. 返回 count + summary

## 自动刷新实现建议

V3.1 不改变 V3.0 的 polling 机制，只增强可观察性。

### 规则

1. 仅 `watchEnabled` 时设置 `setInterval`
2. 若 `isRefreshing === true`，本周期跳过
3. 成功则更新时间
4. 失败则进入 error state，但保留旧数据
5. 下个周期继续尝试

### 不做项

1. 不做指数退避
2. 不做复杂 retry policy 配置
3. 不做 pause/resume UI

原因：

1. V3.1 是小版本 polish
2. 当前 3000ms 重试已经足够

## `/data.json` 错误展示增强

## 分类建议

### `http_500`

识别：

1. `response.ok === false`
2. `response.status === 500`

展示：

1. `Refresh failed`
2. `Server returned 500`
3. `Showing last successful data`

### `json_parse_failed`

识别：

1. `response.ok === true`
2. `JSON.parse` 抛错

展示：

1. `Refresh failed`
2. `JSON parse failed`
3. `Showing last successful data`

### `run_not_found`

识别：

1. `/data.json` 返回的 `selectedRunId` 为空
2. 且当前显式 `--run` 在模型中缺失

展示：

1. `Selected run not found`
2. `Showing last successful data`

### `no_runs`

识别：

1. `runs.length === 0`

展示：

1. empty dashboard
2. 不是 refresh error

### `partial_errors`

识别：

1. `/data.json` 成功
2. `runDetailsById` 内存在 error 字段

展示：

1. `Refreshed with partial errors`
2. 仍更新页面
3. 具体局部错误在对应区域显示

## 与现有 render 流程的关系

V3.1 推荐继续复用现有 render 主流程。

### 原则

1. 不做第二套 page renderer
2. 不改现有 safe view model 主逻辑
3. 只在页面顶层新增 serve runtime status 区
4. 局部错误继续沿用当前 `Detail unavailable / Report unavailable / Workflow unavailable`

这样能最小化回归范围。

## 风险点

## 1. runtime 状态与现有 render 耦合过深

风险：

1. 可能不小心把 serve runtime 逻辑塞进现有 DashboardModel

应对：

1. runtime 状态独立于 DashboardModel
2. 使用 `window.__MOREAGENT_DASHBOARD_RUNTIME__` + 浏览器内存状态

## 2. partial error 判定不一致

风险：

1. 可能把 child error 当作 full refresh failure

应对：

1. 明确 hard error vs partial success 的判断函数

## 3. 手动刷新与自动刷新并发

风险：

1. 用户手动点按钮时正好碰上 watch 轮询

应对：

1. `isRefreshing` guard
2. 并发时跳过第二个请求

## 4. serve startup output 断言过脆

风险：

1. 测试若过度依赖整段文本，会导致小改文案就回归

应对：

1. 断言关键行
2. 不要求整段完全逐字匹配

## 5. 文档更新滞后

风险：

1. V3.1 做了 UI/runtime 增强，但 README / troubleshooting 没跟上

应对：

1. 把 docs update 放进实现步骤和测试清单

## 回滚策略

V3.1 是 usability 增强，回滚应容易。

推荐回滚粒度：

1. 保留 V3.0 serve/watch 主功能
2. 仅移除 runtime status strip
3. 仅移除增强错误文案
4. 仅移除 startup output 增量

不要回滚：

1. V3.0 serve/watch 核心能力
2. selected run persist
3. `/data.json` endpoint

## 最终技术结论

V3.1 的最优实现路径是：

1. 保持 V3.0 的本地服务架构不变
2. 不改 DashboardModel 主 schema
3. 把 usability / observability 状态放进 runtime metadata 和浏览器内存状态
4. 用最小前端脚本增量补上 startup summary、refresh status、error display
5. 用 regression tests 固定这些行为

这足以让 OpenCode 直接开始实现，而不会把 V3.1 做成一次不必要的大重构。
