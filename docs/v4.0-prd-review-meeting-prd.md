# MoreAgent V4.0 PRD: PRD Review Meeting

## 1. 文档目标

本文件定义 MoreAgent V4.0 的产品目标：强化 full workflow 中的 PRD review meeting。

这不是新增一条新 workflow，也不是推翻当前 full workflow。

当前真实源码已经有：

1. `brain -> prd -> prd-review -> prd-gate -> tech-plan -> tech-gate -> implementation -> test -> review`
2. `frontend-prd-review.md`
3. `backend-prd-review.md`
4. `test-prd-review.md`
5. `prd-decision.md`
6. `prd-revision-1.md`
7. `prd-revision-2.md`

V4.0 的工作，是把这段已经存在的流程收紧为稳定、清晰、可恢复、可验收的 PRD review meeting 协议。

## 2. 用户目标

用户在 full workflow 中希望实现的不是“product 写完 PRD 就直接过”，而是：

1. product 先输出 PRD
2. frontend / backend / tester 三个角色共同评审 PRD
3. 每个角色输出结构化问题、风险、缺失信息
4. product 基于这些反馈修订 PRD
5. brain 最终给出 `APPROVED` 或 `CHANGES_REQUESTED`
6. 整个过程能记录到 run、session、artifact、status、report 中
7. 如果中断，后续可以 resume

用户真正要的是“需求阶段的多角色会审闭环”，而不是一个单次 PRD 文件输出。

## 3. 当前 full workflow 中 PRD 阶段真实行为

基于当前真实源码：

### 3.1 当前 phase 结构

来自 `src/workflow.ts`：

1. `prd` phase
   - session: `product`
   - artifact: `prd.md`
2. `prd-review` phase
   - session: `frontend-prd-review`
   - session: `backend-prd-review`
   - session: `test-prd-review`
3. `prd-gate` phase
   - session: `prd-gate-1`
   - artifact: `prd-decision.md`

### 3.2 当前 `start.ts` 的真实执行行为

来自 `src/commands/start.ts`：

1. `prd` phase 先由 `product` 产出 `prd.md`
2. `prd-review` phase 顺序执行：
   - `frontend-prd-review`
   - `backend-prd-review`
   - `test-prd-review`
3. 所有前序 artifact 会累计进入共享的 `artifactContexts`
4. `prd-gate` 由 `brain` 读取上下文并输出 `prd-decision.md`
5. 如果 `prd-decision.md` 是 `Decision: CHANGES_REQUESTED`
   - 最多再进行 2 轮 product revision
   - product 写 `prd-revision-1.md`、`prd-revision-2.md`
   - brain 写 `prd-decision-1.md`、`prd-decision-2.md`

### 3.3 当前已经具备的能力

当前代码并不是没有 PRD review meeting，而是已经有一个基础版：

1. 三个 review 角色已经存在
2. brain gate 已经存在
3. 最多 2 轮修订已经存在
4. `resume` 已经支持 full workflow
5. `status / inspect / report / dashboard` 已能读取 `prd-decision*.md`

### 3.4 当前不足

当前不足不在 phase 缺失，而在协议不够硬：

1. PRD review artifact 只有模板，没有严格结构约束
2. 没有明确规定 review 角色必须写哪些问题类型
3. `frontend-prd-review.md` / `backend-prd-review.md` / `test-prd-review.md` 当前默认模板顶部是 `Decision: CHANGES_REQUESTED`，但这些文件本身不被机器直接判定，容易造成误解
4. 没有明确规定 product 修订时必须回答哪些 review 问题
5. `prd-gate` 是否通过完全依赖 brain 的最终判断，但没有一份清晰的“会议协议文档”
6. 中断和恢复虽然已有实现，但产品文档没有明确对外说明

## 4. V4.0 要解决的问题

V4.0 要解决的不是“怎么跑 phase”，而是“怎么把 PRD 会审做成稳定协议”。

核心问题：

1. review 角色的输出不够结构化
2. product 修订行为不够明确
3. brain gate 的判定依据没有产品化文档
4. 用户无法只看 artifact 就知道 PRD 为什么被打回
5. 刚毕业开发者接手实现时，无法知道哪些文件是生产者，哪些文件是消费者

V4.0 的目标是把这段流程从“代码里已经能跑”提升为“协议清晰、实现稳定、验收明确”。

## 5. PRD review meeting 的参与角色

V4.0 参与角色固定为：

1. `product`
2. `frontend-prd-review`
3. `backend-prd-review`
4. `test-prd-review`
5. `brain / prd-gate`

说明：

1. 这里的 `frontend-prd-review` / `backend-prd-review` / `test-prd-review` 是 MoreAgent session 名
2. 它们对应的 OpenCode 实际 agent 仍然是：
   - `frontend`
   - `backend`
   - `tester`

## 6. 每个角色的输入、输出、禁止事项

## 6.1 `product`

创建阶段：

1. `prd`
2. `prd-gate` 失败后的 revision 轮次

输入：

1. 用户任务
2. `brain-plan.md`
3. 当前 PRD 内容
4. 三个 PRD review artifact
5. 上一轮 `prd-decision*.md`

输出：

1. `prd.md`
2. `prd-revision-1.md`
3. `prd-revision-2.md`

禁止事项：

1. 不修改代码
2. 不跳过 reviewer 提出的问题
3. 不直接输出 gate 决策
4. 不把模糊问题留给后续技术方案阶段兜底

## 6.2 `frontend-prd-review`

创建阶段：

1. `prd-review`

输入：

1. `prd.md`
2. `brain-plan.md`
3. 用户任务上下文

输出：

1. `frontend-prd-review.md`

必须覆盖的问题类型：

1. 前端范围是否清楚
2. 页面/交互/状态是否定义清楚
3. UI 依赖和边界条件是否明确
4. 是否缺失关键前端输入
5. 是否存在无法实施或高风险项

禁止事项：

1. 不修改代码
2. 不直接改写 PRD
3. 不输出最终 gate 结论
4. 不把前端实现方案提前写成 tech-plan

## 6.3 `backend-prd-review`

创建阶段：

1. `prd-review`

输入：

1. `prd.md`
2. `brain-plan.md`
3. 用户任务上下文

输出：

1. `backend-prd-review.md`

必须覆盖的问题类型：

1. API / 数据模型要求是否清楚
2. 服务边界是否清楚
3. 数据约束、权限、错误处理是否缺失
4. 是否存在无法实施或高风险项
5. 是否存在与现有后端边界冲突的需求

禁止事项：

1. 不修改代码
2. 不直接改写 PRD
3. 不输出最终 gate 结论
4. 不提前生成 backend 实施方案

## 6.4 `test-prd-review`

创建阶段：

1. `prd-review`

输入：

1. `prd.md`
2. `brain-plan.md`
3. 用户任务上下文

输出：

1. `test-prd-review.md`

必须覆盖的问题类型：

1. PRD 是否可测
2. 验收标准是否可验证
3. 是否缺少输入输出样例
4. 是否缺少失败场景 / 边界场景
5. 是否存在测试无法落地的空白区

禁止事项：

1. 不修改代码
2. 不直接输出测试结果
3. 不输出最终 gate 结论
4. 不提前写 `test-plan.md`

## 6.5 `brain / prd-gate`

创建阶段：

1. `prd-gate`
2. 每次 PRD revision 之后

输入：

1. `brain-plan.md`
2. `prd.md` 或 `prd-revision-*.md`
3. `frontend-prd-review.md`
4. `backend-prd-review.md`
5. `test-prd-review.md`
6. 上一轮 `prd-decision*.md`

输出：

1. `prd-decision.md`
2. `prd-decision-1.md`
3. `prd-decision-2.md`

禁止事项：

1. 不修改代码
2. 不越权替 product 直接修 PRD
3. 不跳过结构化 review 直接给 through
4. 不输出模糊决策文本

## 7. PRD review artifact 协议

V4.0 需要明确以下文件是正式协议文件。

## 7.1 `frontend-prd-review.md`

文件名：

1. `frontend-prd-review.md`

创建阶段：

1. `prd-review`

生产者：

1. `frontend`

消费者：

1. `product`
2. `brain`

协议要求：

1. 必须包含 review summary
2. 必须包含问题/风险列表
3. 必须明确哪些信息缺失
4. 必须区分“建议项”和“阻塞项”

## 7.2 `backend-prd-review.md`

文件名：

1. `backend-prd-review.md`

创建阶段：

1. `prd-review`

生产者：

1. `backend`

消费者：

1. `product`
2. `brain`

协议要求：

1. 必须包含 feasibility 评估
2. 必须列出 API / data concerns
3. 必须区分阻塞项与非阻塞项

## 7.3 `test-prd-review.md`

文件名：

1. `test-prd-review.md`

创建阶段：

1. `prd-review`

生产者：

1. `tester`

消费者：

1. `product`
2. `brain`

协议要求：

1. 必须包含 testability assessment
2. 必须列出 coverage concerns
3. 必须指出不可验证或歧义的 acceptance criteria

## 7.4 `prd-decision.md`

文件名：

1. `prd-decision.md`
2. `prd-decision-1.md`
3. `prd-decision-2.md`

创建阶段：

1. `prd-gate`

生产者：

1. `brain`

消费者：

1. `start.ts` gate 判定逻辑
2. `report`
3. `status`
4. `dashboard`
5. `resume`

协议要求：

1. 顶部必须存在且仅存在一个 machine-readable decision line
2. 只能是：
   - `Decision: APPROVED`
   - `Decision: CHANGES_REQUESTED`
3. 后文必须说明：
   - 核心结论
   - 已解决问题
   - 剩余问题

## 7.5 `prd-revision-1.md` / `prd-revision-2.md`

文件名：

1. `prd-revision-1.md`
2. `prd-revision-2.md`

创建阶段：

1. PRD gate 第 1 次打回后
2. PRD gate 第 2 次打回后

生产者：

1. `product`

消费者：

1. `brain`
2. 后续 `tech-plan`

协议要求：

1. 必须明确这是 revision 版本
2. 必须回应上一轮 review / gate 问题
3. 不得只复制原始 `prd.md`

## 8. Gate 协议

PRD gate 顶部 machine-readable 字段固定为：

1. `Decision: APPROVED`
2. `Decision: CHANGES_REQUESTED`

行为规则：

1. `APPROVED` => 当前 PRD gate 通过，进入 `tech-plan`
2. `CHANGES_REQUESTED` => 回到 product revision
3. 缺少字段 => 当前实现兼容可视为 unknown，但 V4.0 推荐收紧为 gate 失败

V4.0 推荐产品要求：

1. 对 `prd-decision*.md` 收紧要求
2. 不能依赖模糊自然语言判定是否通过

## 9. 最大修订轮次

当前真实源码：

1. `runPrdGate()` 中 `maxRounds = 2`

V4.0 保持该规则不变：

1. 初始 PRD gate 失败后，最多 2 轮 product 修订
2. 超过 2 轮仍未通过，则 run 在 `prd-gate` 阶段失败

这符合当前路线：

1. V4.0 强化会议协议
2. 不重写 full workflow 主体

## 10. 失败 / 中断 / 恢复行为

## 10.1 失败

失败条件：

1. 任一 `prd-review` session 进程失败
2. `product` revision session 失败
3. `brain` gate session 失败
4. 超过最大修订轮次仍未通过

失败结果：

1. `run.status = failed`
2. `run.workflow.failedPhase = prd-gate` 或 `prd-review`
3. 已完成 session 保留
4. artifact 保留

## 10.2 中断

中断行为遵循当前全局机制：

1. run 可能停留在 `running`
2. 下次 resume 时从未完成 phase 继续

## 10.3 恢复

当前真实源码已经支持：

1. `moreagent start --resume --latest`
2. `moreagent start --resume --run <id>`

V4.0 需要明确产品语义：

1. 若中断发生在 `prd-review`
   - resume 应从 `prd-review` 继续
2. 若中断发生在 `prd-gate`
   - resume 应从 `prd-gate` 继续
3. 已完成 session 不应重复执行

## 11. dashboard / report / status 是否需要展示变化

结论：

1. 不需要新增新的顶层 phase
2. 不需要新增新的 JSON 主 schema
3. 需要确保现有展示继续正确读取 PRD gate 产物

原因：

1. `status` 已读取 `prd-decision*.md`
2. `report` 已汇总 `prdGate`
3. `dashboard` 已显示 `PRD Gate`
4. `inspect --workflow` 已能展示 workflow phases 和 gate

V4.0 推荐的展示要求：

1. `status` 不崩
2. `report` 能正确显示 `prdGate = APPROVED / CHANGES_REQUESTED / unknown`
3. `dashboard` 不崩，继续显示 PRD Gate 卡片
4. 可选增强：
   - 在 future version 中展示最新 PRD review artifacts 摘要

但这不是 V4.0 必做范围。

## 12. CLI 行为

V4.0 不新增新的 CLI 命令。

依赖现有命令：

1. `moreagent start --once --task "..."`
2. `moreagent start --resume --latest`
3. `moreagent start --resume --run <id>`
4. `moreagent start --once --task "..." --from-phase prd-review`
5. `moreagent status`
6. `moreagent inspect --workflow`
7. `moreagent report --latest`
8. `moreagent dashboard`

V4.0 的目标是让这些命令在 PRD review meeting 更严格后仍保持一致可用。

## 13. 用户验收标准

V4.0 满足以下标准即可验收通过：

1. full workflow 的 `prd -> prd-review -> prd-gate` 仍按当前 3-phase 结构执行。
2. `frontend-prd-review.md`、`backend-prd-review.md`、`test-prd-review.md` 都会稳定生成。
3. `product` 在 gate 打回后最多进行 2 轮修订。
4. `brain` 会输出 `Decision: APPROVED` 或 `Decision: CHANGES_REQUESTED`。
5. PRD gate 失败时，run 会在 `prd-gate` 阶段失败，而不是误进入 `tech-plan`。
6. resume from `prd-review` / `prd-gate` 可继续工作。
7. `status / report / dashboard / inspect` 不因 PRD review meeting 强化而崩溃。
8. MVP profile 行为不受影响。

## 14. 最终结论

V4.0 的本质不是“把 PRD 阶段从 0 做出来”，而是：

1. 以当前真实 full workflow 为基线
2. 把现有 PRD review 会审协议明确化
3. 把 review / revision / gate 的生产者和消费者固定下来
4. 让 PRD 阶段具备可恢复、可追踪、可验收的产品级边界

这应当被视为“强化已存在流程”，不是“新增平行系统”。
