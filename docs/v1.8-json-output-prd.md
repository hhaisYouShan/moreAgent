# V1.8 JSON Output PRD

## 目标

为 `moreagent status` 和 `moreagent inspect` 新增 `--json` 输出模式，方便 dashboard、CI、自动化验收和多 agent 调度消费结构化数据。

## 范围

| 命令 | 模式 |
|---|---|
| `moreagent status --json` | JSON 列表（最近 10 条） |
| `moreagent status --latest --json` | JSON 单 run |
| `moreagent status --latest-repair --json` | JSON 单 run |
| `moreagent status --latest-full --json` | JSON 单 run |
| `moreagent status --run <id> --json` | JSON 单 run |
| `moreagent status --run <id> --summary --json` | JSON summary（紧凑） |
| `moreagent inspect --run <id> --json` | JSON 单 run |
| `moreagent inspect --run <id> --workflow --json` | JSON workflow 详情 |

`inspect --agent --json` **暂不支持**，返回 `{ error: { code: "UNSUPPORTED_JSON_MODE" } }`。

## `--json` 输出规则（强制）

1. **stdout 永远输出合法 JSON** — 无论成功或失败。
2. 成功输出 `{ "run": {...} }` 或 `{ "runs": [...] }`。
3. 错误输出 `{ "error": { "code": "...", "message": "..." } }`，`process.exit(1)`。
4. **stderr 不输出业务错误** — stderr 只用于调试日志，避免破坏机器消费。业务错误信息在 JSON `error.message` 中。
5. 非 `--json` 模式保持原有文本输出行为不变。

## 错误处理策略

**选择方案 B：cli.ts 统一 catch 处理 `--json` 错误。**

- `statusCommand` / `inspectCommand` 内部不 try/catch，直接 throw。
- `cli.ts` 的顶层 `catch` 检测 `args.includes('--json')`：
  - 若为 JSON 模式：输出 `{ "error": { "code": "INTERNAL_ERROR", "message": "<err.message>" } }`，`process.exit(1)`。
  - 若非 JSON 模式：保持当前 `console.error` 文本输出。
- 业务级错误（如 "No runs found" / "Run not found"）在命令内部通过 `printJsonError` 处理，返回特定 code。

## JSON Schema

### status --json 列表模式

```json
{
  "runs": [
    {
      "id": "run-xxx",
      "task": "...",
      "status": "completed",
      "profile": "mvp",
      "createdAt": "2026-06-30T...",
      "hasRepair": false
    }
  ]
}
```

列表模式使用轻量 RunSummary（不含 sessions/gates/worktree），只提供索引字段。详情用 `--run <id> --json` 获取。

### Run Summary（status --summary 级）

```json
{
  "run": {
    "id": "run-2026-06-30T02-40-35-8178c8",
    "task": "...",
    "status": "completed",
    "createdAt": "2026-06-30T02:40:35.310Z",
    "profile": "full",
    "completed": true,
    "failed": false,
    "canResume": false,
    "resumeBlockedReason": "run is completed",
    "canMerge": true,
    "mergeBlockedReason": "",
    "hasRepair": false,
    "repairSessions": { "total": 0, "failureAnalysis": 0, "repairs": 0, "retries": 0, "revisions": 0 },
    "hasGate": true,
    "gates": { "prdGate": "APPROVED", "techGate": "APPROVED", "test": "PASS", "review": "APPROVED" },
    "worktree": {
      "hasWorktree": true,
      "exists": true,
      "path": "/path/to/worktree",
      "branch": "agent/run-xxx",
      "dirty": false
    },
    "lastFailed": null,
    "nextActions": ["moreagent merge --run run-xxx"]
  }
}
```

### Run Detail（status --latest 级）

Summary 基础上嵌套 sessions 数组：

```json
{
  "run": {
    "...": "...",
    "completedPhases": ["brain", "prd", "..."],
    "failedPhase": null,
    "sessions": [
      {
        "agentName": "brain",
        "status": "completed",
        "durationSeconds": 32,
        "startedAt": "...",
        "completedAt": "...",
        "artifactDir": "...",
        "runtimeSessionId": "ses_xxx",
        "worktreePath": "...",
        "error": null
      }
    ]
  }
}
```

- `durationSeconds`: `number | null` — 文本输出保持 `"32s"` / `"N/A"`，JSON 用数值。

### Workflow Detail（inspect --workflow 级）

```json
{
  "run": { "...": "..." },
  "workflow": {
    "profile": "full",
    "currentPhase": null,
    "completedPhases": ["brain", "prd", "..."],
    "failedPhase": null,
    "gates": { "...": "..." },
    "repairSessions": [
      { "agentName": "repair-1-implementer", "status": "completed" }
    ],
    "lastFailed": null,
    "suggestedCommands": ["moreagent inspect --run ...", "..."]
  }
}
```

### 错误 JSON

```json
{
  "error": {
    "code": "NO_RUNS",
    "message": "No runs found."
  }
}
```

错误码定义：

| code | 场景 |
|---|---|
| `NO_RUNS` | sessions.json 为空 |
| `RUN_NOT_FOUND` | --run 指定的 id 不存在 |
| `NO_REPAIR_RUN` | --latest-repair 无结果 |
| `NO_FULL_RUN` | --latest-full 无结果 |
| `NOT_INITIALIZED` | .moreagent 不存在 |
| `INTERNAL_ERROR` | 未预期的异常（cli.ts catch） |
| `UNSUPPORTED_JSON_MODE` | inspect --agent --json 暂不支持 |

## 呈现模型

当前 `status.ts` 和 `inspect.ts` 直接在 console.log 中渲染格式化字符串。V1.8 应内部提取纯函数 `buildRunSummary(run): RunSummary`，返回纯数据对象，然后：
- 文本渲染器：用当前逻辑格式化输出
- JSON 渲染器：`printJson(data)` 输出 `JSON.stringify(data, null, 2)`

## 验收标准

1. 所有现有文本输出不变
2. `--json` 下 stdout 永远输出合法 JSON（成功或错误）
3. 错误场景 exit code = 1，stderr 无业务错误
4. `status --json` 列表模式可 parse
5. `status --latest --json` 可 parse
6. `status --run missing --json` 输出 `{ error: { code: "RUN_NOT_FOUND" } }` + exit 1
7. `inspect --run missing --json` 输出 `{ error: { code: "RUN_NOT_FOUND" } }` + exit 1
8. 非 JSON 模式错误输出保持原文本行为
9. MVP run 和 full workflow run 均能正常输出 JSON
10. schema 字段名首版确定后不再改名
