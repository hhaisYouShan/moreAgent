# V2.1 Dashboard 可读性增强 Technical Plan

## 目标

在不改变 Dashboard 架构的前提下，为 V2.1 增强 Dashboard 的可读性和信息层次。

这不是功能扩张版本，而是一个“小而实用”的前端表现层增强版本。

## 路线调整说明

V2.1 的定位相对 V2.0 tech plan 做了明确收敛。

原先路线中，Dashboard 后续演进曾包含：

1. `dashboard --serve`
2. `dashboard --watch`
3. 自动刷新

V2.1 现明确调整为 usability enhancement，而不是继续推进运行模式扩展。

调整原因：

1. V2.0 / V2.0.1 刚稳定，优先级应先放在读得更清楚，而不是跑得更复杂。
2. 静态 Dashboard 已经具备完整的数据面，当前短板是信息层次、状态辨识度和解释能力。
3. server / watch / 自动刷新会引入额外的生命周期管理、测试负担和边界条件，不适合作为刚稳定后的下一步。

新的版本路线：

1. V2.1：Dashboard 可读性增强
2. V2.2：Dashboard 空状态 / 错误态 / 数据稳定性补强
3. V2.3：`dashboard --open`
4. V3.0：`dashboard --serve` / `--watch` / 自动刷新

## 改动范围

V2.1 只涉及 Dashboard 的表现层和少量 view helper 设计。

建议改动范围：

- `src/commands/dashboard.ts`
- 可选新增极少量 dashboard 内部 helper
- `CHANGELOG.md`

本次文档任务不实现代码，只定义边界。

## 不改动范围

V2.1 不改：

1. `src/commands/report.ts` 的 report 逻辑
2. JSON schema
3. `status --json` / `report --json` / `inspect --workflow --json`
4. `dashboard --run / --limit / --output` 参数行为
5. test fixture 语义
6. package.json
7. 外部依赖

## 数据来源

V2.1 仍只消费现有 Dashboard model。

不新增数据采集命令，不新增 schema，不直接读取 sessions.json。

继续使用 V2.0 的模型边界：

```ts
interface DashboardModel {
  generatedAt: string;
  selectedRunId: string | null;
  runs: RunListItem[];
  runDetailsById: {
    [runId: string]: {
      status: StatusRunDetail | null;
      statusError?: DashboardError;
      report: WorkflowReport | null;
      reportError?: DashboardError;
      workflow: WorkflowInspect | null;
      workflowError?: DashboardError;
    };
  };
}
```

结论：

- V2.1 是纯渲染层升级
- 不需要改数据层协议

## HTML / CSS / JS 修改点

## 1. Sidebar Run List

### HTML

建议增强每个 run item 的结构：

- 顶部：task 摘要
- 次级：run id
- 底部 meta 行：
  - status badge
  - recommendation badge
  - profile badge
  - createdAt short text

### CSS

增加更清晰的视觉层级：

- failed：红色边界或红点
- running：黄/橙色边界或黄点
- merge ready：绿色 recommendation
- selected：更明确的背景/边界高亮

### JS

不新增数据逻辑。

只在渲染时根据：

- `run.status`
- `report.report.decision.recommendation`
- `report.report.decision.overallStatus`
- `run.profile`

输出不同 class。

## 2. Enhanced Summary 区域

### HTML

新增或重构顶部 summary block，拆成：

1. 主结论区
2. 条件状态区

主结论区字段：

- run id
- task
- overallStatus
- recommendation

条件状态区字段：

- canResume
- canMerge
- mainClean
- worktree exists

### CSS

- overallStatus 使用最大字号或最强 badge
- recommendation 使用次级高亮 badge
- 条件字段使用 compact status pills

### JS

需要新增轻量 helper，把 report 数据映射为 summary cell 列表。

## 3. Workflow Progress 区域

### HTML

full workflow：

- 固定渲染 9 phases
- phase 顺序固定

MVP run：

- 不渲染 9-phase bar
- 改为 banner：
  - `workflow unavailable`
  - `MVP run`

### CSS

phase 样式至少区分：

- completed
- failed
- pending

可继续沿用 V2.0 的 pill 模型，但颜色和对比度要更明确。

### JS

建议新增 helper：

- `buildWorkflowPhaseViewModel(detail)`

负责把 workflow 信息映射为：

- phaseName
- phaseState
- current / failed / completed

MVP run 直接返回降级标记。

## 4. Gate / Test / Review 卡片

### HTML

保留四卡片结构：

- PRD Gate
- Tech Gate
- Test
- Review

### CSS

统一状态色规则：

- green: `APPROVED` / `PASS`
- red: `CHANGES_REQUESTED` / `FAIL`
- neutral-warning: `unknown`

### JS

建议新增 helper：

- `getDecisionTone(value)`

统一输出：

- success
- danger
- neutral

避免 gate/test/review 分散判断。

## 5. Merge Readiness 解释区

### HTML

在现有 merge readiness 区上方增加 explanation block：

- `Ready reason`
- 或 `Blocked reason`

显示摘要解释，不要求复杂推理图。

### 解释来源

继续完全来自现有 report model：

- `recommendation`
- `canMerge`
- `mainClean`
- `worktree.exists`
- `blockedReason`

### JS

建议新增 helper：

- `buildMergeExplanation(report)`

输出：

- title
- reason list

例如：

- `MERGE_READY because run passed, main is clean, and worktree exists`
- `BLOCKED because main repository has uncommitted changes`

## 6. JSON / Debug 区

### HTML

从“默认占据主内容区块”调整为：

- 默认折叠
- 展开后仍保留 tabs

### CSS

- 折叠态为低优先级区块
- 展开态保持现有 JSON readability

### JS

新增简单的 disclosure 交互即可：

- open / close debug section

不需要引入复杂状态管理。

## 是否需要新增 helper

建议新增，但保持最少：

1. `formatSidebarRunMeta(run, detail)`
2. `buildSummaryViewModel(detail)`
3. `buildWorkflowPhaseViewModel(detail)`
4. `getDecisionTone(value)`
5. `buildMergeExplanation(report)`
6. `formatTaskPreview(task)`

这些 helper 应只服务于渲染层，不改变数据层。

## 测试计划

V2.1 应补充或增强 Dashboard 相关测试，至少覆盖以下场景。

### 1. Enhanced Summary 区存在

检查 HTML 包含新的 summary 区标识，例如：

- `Overall Status`
- `Recommendation`
- `Can Resume`
- `Can Merge`
- `Main Clean`
- `Worktree Exists`

### 2. failed run 在 sidebar 有明显标记

检查 failed run item 带有明确 class 或文本标记，能够与普通 completed 区分。

### 3. MERGE_READY 显示 ready reason

检查 HTML 中包含 ready explanation，而不仅是 badge。

### 4. BLOCKED 显示 blocked reason

检查 HTML 中包含 blocked explanation，尤其针对：

- dirty main
- missing worktree
- `canMerge = false`

### 5. MVP run 仍显示 workflow unavailable

检查非 full workflow run 不回退，并继续显示：

- `workflow unavailable`
- `MVP run`

### 6. JSON / Debug 区仍可访问

检查 HTML 中仍存在：

- status tab
- report tab
- workflow tab

即使默认折叠，也必须可展开访问。

### 7. dashboard --limit / --run / --output 行为不变

回归确认：

- `--limit` 仍控制预取数量
- `--run` 仍控制默认选中 run
- `--output` 仍控制写入路径

### 8. 原有 dashboard hardening 测试不回退

必须保证 V2.0.1 的 hardening 语义不回退，尤其包括：

- `window.__MOREAGENT_DASHBOARD_DATA__` 安全序列化
- 参数校验
- HTML 结构可解析

## 风险点

### 1. 视觉增强过度，导致 DOM 结构不稳定

如果一次性重做太多结构，现有 dashboard regression test 可能大量波动。

建议：

- 保持区域划分不变
- 优先做信息层次增强，而不是大改布局

### 2. 解释文案与真实状态不一致

Merge explanation 和 summary explanation 如果自己再做一套推理，容易与 report 判定不一致。

建议：

- 所有 explanation 都基于现有 report model
- 不重新定义 recommendation 逻辑

### 3. Sidebar 信息过多

如果在 run list 中塞过多字段，会损害扫读效率。

建议：

- task 只保留单行摘要
- createdAt 用短格式
- 主打 badge 和状态点

### 4. JSON/debug 折叠后被误删可访问性

如果只顾“简洁”，可能会让 debug 信息变得难找。

建议：

- 默认折叠
- 但入口明显

## 不做项说明

V2.1 明确不做：

1. 本地 server
2. `--serve`
3. `--watch`
4. 自动刷新
5. merge / resume / start 按钮
6. React / Vue
7. 外部 UI 库
8. 复杂图表
9. Dashboard 数据模型改造
10. report/status JSON schema 变更

## 是否建议进入实现

建议进入实现。

原因：

1. V2.0 功能面已经稳定，V2.1 不需要再补新能力。
2. 当前提升点集中在可读性，范围小、收益直接。
3. 技术边界清晰，不需要动 CLI 数据协议。
4. 风险主要在渲染层，可控。
