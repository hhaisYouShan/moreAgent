# V3.1 Dashboard Serve / Watch Usability PRD

## 背景

假设 V3.0 已完成以下能力：

1. `moreagent dashboard --serve`
2. `moreagent dashboard --serve --watch`
3. `moreagent dashboard --serve --open`
4. `GET /`
5. `GET /data.json`
6. `GET /health`
7. serve 模式下手动 `Refresh data`
8. watch 模式下 3000ms polling

V3.0 解决的是“能用”：

1. 本地服务能启动
2. 页面能打开
3. 数据能刷新

但还没有系统解决“好不好用”：

1. 启动后 CLI 给用户的信息是否足够清楚
2. 页面上是否能看出当前在刷新、刚刷新过、刷新失败、失败后是否还保留旧数据
3. `/data.json` 出错时用户能否判断问题是 server 层、JSON 层、run 层、还是 status/report/inspect 的局部错误
4. README / docs / troubleshooting 是否能支撑 serve 模式日常使用

因此 V3.1 不引入大功能，不改变产品边界，而是只做 serve/watch 模式的可用性、可观测性和用户体验增强。

## V3.1 目标

V3.1 的目标是让 serve/watch 模式达到“长期开着看也不迷糊”的可用度。

具体目标：

1. CLI 启动输出更清楚
2. 页面刷新状态更可见
3. `/data.json` 错误展示更可解释
4. 用户文档对 serve 模式更完整
5. 测试覆盖这些可用性行为，避免后续回退

## 非目标

V3.1 明确不做：

1. 远程服务
2. 登录
3. 多人协作
4. merge / resume / start / clean 等执行型按钮
5. React / Vue
6. Express
7. WebSocket
8. SSE
9. 改造 Dashboard 为独立前端工程
10. 新的 workflow / run 管理功能
11. 修改 DashboardModel 主体 schema，除非有明确兼容策略

## 用户问题

## 1. serve 启动后，用户不知道当前服务到底怎么跑的

当前仅打印 URL 不够，用户还需要知道：

1. watch 是否开启
2. refresh interval 是多少
3. 当前选中了哪个 run
4. limit 是多少
5. host / port 是什么

否则用户常见疑问会变成：

1. “为什么没自动刷新”
2. “我现在是不是 watch 模式”
3. “为什么 selected run 不是我想看的那个”
4. “是不是只加载了 10 条”

## 2. 页面刷新行为不可观察

如果页面自动刷新或手动刷新，用户至少要知道：

1. 是否正在刷新
2. 上次刷新成功是什么时候
3. 上次刷新失败了没有
4. 失败后当前页面显示的是旧数据还是空数据

否则 watch 模式会让用户无法判断：

1. 页面是静止的
2. 还是刷新失败了
3. 还是没有新数据

## 3. `/data.json` 错误不够可解释

serve/watch 模式的错误来源可能有多层：

1. HTTP 500
2. JSON parse failed
3. selected run 找不到
4. no runs
5. status 子结果错误
6. report 子结果错误
7. inspect workflow 子结果错误

如果错误只显示成一个泛化文案，用户会不知道下一步该跑哪个命令。

## 4. 文档还不够把 serve/watch 模式讲清楚

serve mode 相比静态 dashboard 新增了：

1. host / port
2. watch / polling
3. refresh state
4. open URL
5. `/health`
6. `/data.json` error behavior

这部分如果文档不补，用户会：

1. 回到看源码
2. 回到试错
3. 回到问“这个状态是什么意思”

## 产品范围

V3.1 只做 serve/watch 体验增强。

### 1. CLI 启动信息增强

当执行：

```bash
moreagent dashboard --serve
moreagent dashboard --serve --watch
moreagent dashboard --serve --run <id> --limit 20 --open
```

CLI 应在服务启动成功后输出一个清晰的 serve summary。

至少包含：

1. URL
2. host
3. port
4. watch 是否开启
5. refresh interval
6. selected run
7. limit

推荐输出形态：

```text
Dashboard server started
URL: http://127.0.0.1:4317/
Host: 127.0.0.1
Port: 4317
Watch: enabled
Refresh interval: 3000ms
Selected run: run-2026-06-30T12-00-00-abc123
Limit: 10
```

如果未指定 selected run：

```text
Selected run: latest
```

如果未启用 watch：

```text
Watch: disabled
Refresh interval: manual
```

### 2. 页面刷新状态增强

页面顶部或明显区域应新增一个 serve runtime status 区。

该区域至少展示：

1. `Last refreshed`
2. `Refreshing...`
3. `Refresh failed`
4. `Retrying...` 或下次自动重试说明
5. `Showing last successful data`

要求：

1. 手动刷新和自动刷新共享这一状态展示区
2. 刷新失败后不清空旧数据
3. 页面要明确说明当前仍展示旧数据

### 3. `/data.json` 错误展示增强

V3.1 要让页面区分至少以下错误类型：

1. HTTP 500
2. JSON parse failed
3. run not found
4. no runs
5. status 局部错误
6. report 局部错误
7. inspect 局部错误

展示目标不是做复杂错误中心，而是让用户看到：

1. 错误发生在哪一层
2. 当前页面还能不能继续看
3. 下一步应该跑什么 CLI

### 4. 文档体验增强

V3.1 应明确评估并更新：

1. `README.md`
2. `docs/dashboard-usage.md`
3. `docs/troubleshooting.md`
4. `docs/index.md`

产品要求：

1. README 要有简洁 serve/watch 入口
2. dashboard-usage 要写清 runtime status 和 refresh 行为
3. troubleshooting 要新增 serve/watch 错误说明
4. docs index 要能引导用户找到 serve/watch 文档

## 页面行为设计

## Serve Runtime Status 区

推荐位置：

1. main area 顶部
2. Enhanced Summary 上方或同层

推荐字段：

1. mode: `serve`
2. watch: `enabled` / `disabled`
3. refresh interval: `3000ms` / `manual`
4. last refreshed: timestamp
5. refresh state: `idle` / `refreshing` / `error`
6. selected run source: `latest` / explicit run id
7. stale-data note: only when refresh failed

### 状态规则

#### idle

说明：

1. 当前未在请求
2. 上次请求已成功

显示：

1. `Last refreshed: 12:30:15`

#### refreshing

说明：

1. 正在调用 `/data.json`

显示：

1. `Refreshing...`
2. 手动刷新按钮 disabled

#### error

说明：

1. 最近一次刷新失败

显示：

1. `Refresh failed`
2. 错误摘要
3. `Showing last successful data`
4. 如果 watch 开启，显示 `Will retry automatically`

## 手动刷新行为

### 按钮名称

继续使用：

1. `Refresh data`

### 点击行为

1. 按钮进入 disabled
2. 状态区显示 `Refreshing...`
3. 请求 `/data.json`
4. 若成功：
   - 更新页面数据
   - 更新时间
   - 清除错误状态
5. 若失败：
   - 恢复按钮可点击
   - 保留旧数据
   - 显示失败状态

### watch 模式中的手动刷新

允许。

原因：

1. 用户可能不想等下一个 3000ms 周期
2. 自动刷新失败时用户需要手动重试

## 自动刷新行为

仅在 `--watch` 下启用。

推荐规则：

1. 每 3000ms 触发一次
2. 如果当前已经在刷新，则跳过新的轮询周期
3. 失败后保留旧数据
4. 下一个周期自动重试
5. 页面显示最近失败状态，直到下次成功

## `/data.json` 错误行为设计

## A. HTTP 500

页面显示：

1. `Refresh failed`
2. `Server returned 500`
3. `Showing last successful data`

推荐下一步文案：

1. `Check moreagent dashboard server logs or rerun the command`

## B. JSON parse failed

页面显示：

1. `Refresh failed`
2. `JSON parse failed`
3. `Showing last successful data`

推荐下一步文案：

1. `Check /data.json directly and inspect CLI JSON output`

## C. run not found

页面显示：

1. `Selected run not found`
2. `The requested run is no longer available`

推荐行为：

1. 如果是显式 `--run` 场景，保留错误提示，不自动切换 selected run
2. 页面继续显示上一次成功数据

## D. no runs

页面显示：

1. empty dashboard
2. `No runs found`
3. 如果此前有旧数据而刷新后变成 no runs，仍显示 empty state，但不应伪装成刷新错误

## E. status/report/inspect 局部错误

页面应区分：

1. 全局 refresh 成功
2. 但某个 run detail 子块失败

例如：

1. status 局部错误 -> `Detail unavailable`
2. report 局部错误 -> `Report unavailable`
3. inspect workflow 局部错误 -> `Workflow unavailable`

V3.1 目标：

1. 在刷新状态区显示：
   - `Refreshed with partial errors`
2. 不把整个刷新视为完全失败

## CLI 行为

V3.1 不新增新的 serve/watch 命令，仅优化现有输出。

### `moreagent dashboard --serve`

新增输出信息：

1. URL
2. host
3. port
4. watch disabled
5. refresh interval manual
6. selected run
7. limit

### `moreagent dashboard --serve --watch`

新增输出信息：

1. URL
2. host
3. port
4. watch enabled
5. refresh interval 3000ms
6. selected run
7. limit

### `moreagent dashboard --serve --open`

新增要求：

1. 先打印 serve summary
2. 再打印 open success / open failed

### `moreagent dashboard --serve --run <id> --limit <n>`

新增要求：

1. 启动输出里要明确 selected run
2. 如果 selected run 是额外 include 的，也不需要特别报错
3. 只要最终模型包含该 run 即可

## 验收标准

V3.1 产品层验收至少包括：

1. serve 启动输出包含 URL / watch / refresh interval / selected run / limit / host / port
2. 非 watch 模式页面存在手动 `Refresh data`
3. watch 模式页面存在自动轮询且保留手动刷新
4. 页面能显示 `Last refreshed`
5. 页面能显示 `Refreshing...`
6. 页面能显示 `Refresh failed`
7. 刷新失败后明确显示 `Showing last successful data`
8. `/data.json` 的 500 / parse / no runs / partial error 能以可区分方式呈现
9. selected run 在 watch 刷新后不丢失
10. static dashboard 不回退
11. `--open` 行为不回退

## 不做项说明

V3.1 不做：

1. dashboard 内执行 merge/resume/start
2. 服务端主动触发 run
3. 日志流式推送
4. 实时多端协作
5. 复杂错误中心页面
6. 新的远程 API

## 对 OpenCode 的产品结论

V3.1 的正确方向不是“再加新功能”，而是把 V3.0 已有 serve/watch 模式做成：

1. 启动时信息清楚
2. 运行时状态可见
3. 刷新失败后不慌
4. 文档能支撑日常使用

这属于 usability / observability polish，不改变产品边界。
