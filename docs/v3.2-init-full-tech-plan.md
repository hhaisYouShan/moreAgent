# MoreAgent V3.2 Tech Plan: `moreagent init --full`

## 1. 文档目标

本文件给 OpenCode 提供可直接执行的实现施工图，覆盖源码修改点、测试设计、回归点和验收 checklist。

本文件不定义抽象愿景，只定义基于当前真实实现的增量改造。

## 2. 当前源码基线

## 2.1 已有实现

当前 `src/commands/init.ts` 已有：

1. `InitProfile = 'mvp' | 'full'`
2. `MVP_CONFIG`
3. `FULL_CONFIG`
4. `MVP_OPENCODE_AGENTS`
5. `FULL_OPENCODE_AGENTS`
6. `initCommand(profile)`
7. `ensureOpenCodeAgents(profile)`

当前 `src/cli.ts` 已有：

1. `init` 命令分发
2. `--profile full` 解析
3. 默认 profile = `mvp`

当前辅助能力：

1. `src/config.ts` 负责 `.moreagent` 路径与 config 读取
2. `src/queue.ts` 负责初始化 `tasks.json`
3. `src/runtimeSessions.ts` 负责初始化 `runtime-sessions.json`

## 2.2 当前缺失能力

当前没有：

1. `--full` 参数
2. full init 的专用输出文案
3. integration guide 生成逻辑
4. 创建结果的逐项 created/skipped 汇总
5. help / README 中的完整 full init 说明
6. 回归测试覆盖 `init --full`

## 3. 实现策略

## 3.1 总体原则

1. 不改变当前 `init` 的默认 `mvp` 语义
2. `--profile full` 保持兼容
3. `--full` 作为更高层、更明确的 full init 入口
4. 只增加最小必要逻辑，不引入新依赖
5. 保持“只创建缺失文件”的安全默认

## 3.2 推荐设计

引入一个新的 init mode 概念，但不需要大改类型系统：

1. profile 仍然只有 `mvp | full`
2. 新增一个布尔语义 `fullBootstrap`
3. 当 `--full` 或 `--profile full` 时，都进入 full 生成路径

推荐原因：

1. 避免把 profile 和行为入口拆成两个不一致体系
2. 仍然保留最小代码改动

## 4. 需要修改的源码文件

## 4.1 `src/cli.ts`

需要修改。

修改点：

1. `printHelp()` 中加入：
   - `moreagent init --profile full`
   - `moreagent init --full`
2. `init` 分支中新增 `--full` 参数解析
3. 增加冲突参数校验：
   - `--profile mvp --full` => 报错
4. 将解析后的结果传给 `initCommand`

建议实现方式：

1. 保留 `InitProfile`
2. 为 `initCommand` 新增第二个 options 参数，避免继续把所有语义挤进单个 profile 字段

示例接口：

```ts
initCommand(profile, { fullBootstrap: boolean })
```

需要新增：

1. 参数错误文案
2. help 示例

## 4.2 `src/commands/init.ts`

需要修改，是本次主文件。

修改点：

1. 扩展 `initCommand` 签名，支持 `fullBootstrap`
2. 抽出“创建文件/目录并记录 created/skipped”的 helper
3. 抽出 full init 生成逻辑
4. 新增 integration guide 内容生成函数
5. 在重复执行时输出更精确的结果汇总

建议新增 helper：

1. `ensureDir(dir): InitActionResult`
2. `ensureFile(path, content): InitActionResult`
3. `ensureOpenCodeAgents(profile): InitActionResult[]`
4. `buildIntegrationGuide(): string`
5. `ensureIntegrationGuide(): InitActionResult`
6. `printInitSummary(results, profile, mode): void`

建议新增结果类型：

```ts
type InitActionStatus = 'created' | 'skipped';

interface InitActionResult {
  path: string;
  kind: 'file' | 'dir';
  status: InitActionStatus;
  reason?: string;
}
```

这样做的目的：

1. 避免继续用零散 `console.log`
2. 让测试可以断言输出更稳定
3. 保持实现仍然简单

## 6. CLI 行为施工细节

## 6.1 参数优先级

推荐规则：

1. 无参数 => `profile = mvp`, `fullBootstrap = false`
2. `--profile full` => `profile = full`, `fullBootstrap = true`
3. `--full` => `profile = full`, `fullBootstrap = true`
4. `--profile full --full` => 允许，结果仍为 full
5. `--profile mvp --full` => 报错退出

## 6.2 成功输出

建议输出结构：

```text
Initializing MoreAgent project (profile: full, mode: full-bootstrap)...

Created:
  .moreagent/config.yaml
  .moreagent/sessions.json
  .moreagent/tasks.json
  .moreagent/runtime-sessions.json
  .opencode/agents/brain.md
  ...
  .moreagent/integration-guide.md

Skipped:
  .opencode/agents/tester.md (already exists)
  .moreagent/config.yaml (already exists)

Next:
  1. Review .moreagent/config.yaml
  2. Review .moreagent/integration-guide.md
  3. Run: moreagent start --once --task "..."
```

原因：

1. 当前 init 输出不区分 created / skipped
2. `--full` 必须让用户一眼知道自己拿到了什么

## 7. integration guide 内容施工建议

推荐由一个纯字符串 builder 生成，不引入模板引擎。

建议函数：

```ts
function buildIntegrationGuide(): string
```

内容来源：

1. 当前 full profile 角色
2. 当前 full workflow phase 清单
3. 当前仓库的标准命令
4. 当前 `.moreagent` 目录说明

这样做的好处：

1. 无需新增依赖
2. 易于测试
3. 内容变更集中在一个文件内

## 13. 风险点和回归点

### 风险 1：`--profile full` 与 `--full` 语义漂移

风险：

1. 两个入口生成不同内容

控制方式：

1. 共享同一套 full bootstrap 逻辑

### 风险 2：覆盖用户自定义 agent

风险：

1. 再次 init 时误覆盖 `.opencode/agents/*.md`

控制方式：

1. 保持 exists => skip
2. 测试覆盖自定义 agent 不被覆盖

### 风险 3：当前早返回逻辑吞掉后续输出

风险：

1. 当前 `configExists()` 下会早返回，可能导致 full bootstrap 的 guide 创建逻辑不执行

控制方式：

1. 重构 init 流程顺序，把“ensure missing files”统一放到早返回之前
2. 仅在所有 ensure 完成后再打印 next steps

### 风险 4：README 与真实行为不一致

风险：

1. 文档写了 `--full`，但 help 或输出没同步

控制方式：

1. help / README / CHANGELOG 同步纳入验收

### 风险 5：集成说明文档内容与真实 workflow 漂移

风险：

1. guide 写死的 phases / roles 与源码不一致

控制方式：

1. 优先从共享常量或当前 full profile 信息构造 guide
2. 避免手写第二套独立真相
