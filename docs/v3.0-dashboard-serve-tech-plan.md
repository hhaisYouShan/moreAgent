# V3.0 Dashboard `--serve / --watch / auto-refresh` Technical Plan

## 目标

在不破坏 V2.x 静态 Dashboard 的前提下，为 `moreagent dashboard` 增加本地服务模式：

1. `--serve`
2. `--watch`
3. `--serve --open`

实现原则：

1. 不新增外部依赖
2. 尽量复用现有 `buildDashboardModel()` 与 `renderDashboardHtml()`
3. 保持 CLI 心智稳定
4. 保持 Dashboard 仍然是只读视图

## 当前实现现状

根据当前 `src/commands/dashboard.ts` 和 `src/cli.ts`：

1. `dashboardCommand(options)` 已支持 `run / limit / output / open`
2. `buildDashboardModel(selectedRun, limit)` 负责预取最近 runs 和每个 run 的 detail/report/workflow
3. `renderDashboardHtml(model)` 生成完整 HTML
4. 页面 JS 已具备本地切换 run、safe view model、debug 展示能力
5. `--open` 已具备“生成成功后再打开”的稳定语义

这说明 V3.0 不需要重做 Dashboard 数据模型，只需要扩展交付方式。

## 技术设计总原则

### 1. 静态模式与服务模式并存

保留：

1. `dashboard` -> 静态 HTML
2. `dashboard --open` -> 静态 HTML + 打开文件

新增：

1. `dashboard --serve` -> 启动本地 HTTP 服务
2. `dashboard --serve --watch` -> 服务 + 自动刷新数据

### 2. 不引入 Express

推荐使用 Node 内置 `http` 模块。

原因：

1. 路由非常简单
2. 不需要中间件系统
3. 不需要新增依赖
4. 启动更轻
5. 测试更直接

### 3. 页面继续复用当前 HTML/JS 结构

推荐复用：

1. `buildDashboardModel()`
2. `renderDashboardHtml()`

但要做一处结构升级：

1. `renderDashboardHtml(model, options?)`
2. 允许注入“serve 模式配置”，例如：
   - `serveMode`
   - `watchEnabled`
   - `dataEndpoint`
   - `refreshIntervalMs`

## CLI 参数设计

## 建议新增参数

在 `DashboardOptions` 增加：

```ts
export interface DashboardOptions {
  run?: string;
  limit?: number;
  output?: string;
  open?: boolean;
  serve?: boolean;
  watch?: boolean;
  port?: number;
  host?: string;
}
```

## 参数语义

### `--serve`

表示：

1. 不再只生成静态文件
2. 改为启动本地服务

### `--watch`

表示：

1. 在服务模式下启用自动刷新
2. 单独使用 `dashboard --watch` 应视为参数错误

### `--port`

表示：

1. 服务监听端口
2. 仅在 `--serve` 下有效

### `--host`

表示：

1. 服务监听 host
2. MVP 建议只接受 `127.0.0.1` 或 `localhost`

## CLI 行为矩阵

### `moreagent dashboard`

执行：

1. `buildDashboardModel()`
2. `renderDashboardHtml()`
3. `writeDashboardHtml()`
4. exit

### `moreagent dashboard --open`

执行：

1. 静态生成
2. `openInDefaultBrowser(target)`
3. 这里的 target 是 HTML file path
4. exit

### `moreagent dashboard --serve`

执行：

1. 校验 host/port/serve 参数
2. 若传入 `--output`，打印：
   - `--output only applies to static dashboard generation`
3. 构建初始 model
4. 启动 HTTP server
5. 打印 URL
6. 常驻进程

### `moreagent dashboard --serve --open`

执行：

1. 启动 HTTP server
2. 监听成功后 `openInDefaultBrowser(target)`
3. 这里的 target 是 `http://host:port/`
4. open 失败仅打印提示
5. 服务继续运行

### `moreagent dashboard --serve --watch`

执行：

1. 启动 HTTP server
2. 页面默认保留 `Refresh data` 按钮
3. 仅 `--watch` 启用 `3000ms` polling `/data.json`
4. 服务端每次请求动态构建最新 model

### `moreagent dashboard --serve --run <id>`

执行：

1. 初始 model 按既有匹配逻辑选择 run
2. `selectedRunId` 进入页面初始状态
3. 即使该 run 不在 latest `limit` 范围内，也要额外 include 到 DashboardModel
4. `/data.json` 的后续刷新也必须继续 include 这个 selected run
5. 避免 watch 过程中 `selectedRunId` 丢失

### `moreagent dashboard --serve --limit 20`

执行：

1. 数据模型仅预取最近 20 个 runs
2. `/data.json` 每次刷新也保持相同 limit
3. 若指定 `--run <id>`，模型变为：
   - latest N runs
   - plus selected run if not already included

### `moreagent dashboard --serve --output /tmp/dash.html`

推荐语义：

1. 允许传入，但在 `--serve` 下忽略
2. CLI 必须打印说明：
   - `--output only applies to static dashboard generation`

要求：

1. 不写静态 HTML 文件
2. 不把 `--output` 当作 web root
3. 若需要静态文件，继续使用无 `--serve` 模式

## 默认值

推荐默认值：

```ts
const DEFAULT_DASHBOARD_LIMIT = 10;
const DEFAULT_DASHBOARD_OUTPUT = '.moreagent/dashboard/index.html';
const DEFAULT_DASHBOARD_HOST = '127.0.0.1';
const DEFAULT_DASHBOARD_PORT = 4317;
const DEFAULT_REFRESH_INTERVAL_MS = 3000;
```

理由：

1. `4317` 与现有前端 dev 常用端口冲突概率较低
2. `3000/5173/5174` 等端口更常被本机其他项目占用
3. `3000ms` polling 足够看状态变化，不会太频繁

## 服务端设计

## 路由设计

服务端仅提供：

### `GET /`

返回：

1. HTML
2. `Content-Type: text/html; charset=utf-8`

实现建议：

1. 每次请求都基于当前最新 model 动态渲染 HTML
2. 不使用服务启动时的一次性 HTML 快照
3. HTML 仍内嵌当前请求时刻的最新数据

推荐：

1. 首屏直接返回最新 HTML
2. 避免首页首帧过期
3. 保持与静态模式同构

### `GET /data.json`

返回：

1. DashboardModel JSON
2. `Content-Type: application/json; charset=utf-8`

实现建议：

1. 每次请求都重新调用 `buildDashboardModel(run, limit)`
2. 不做进程内长期缓存

原因：

1. 当前模型构建成本可接受
2. 避免复杂 cache invalidation
3. 更接近 CLI 当前行为
4. 更适合 `Refresh data` 和 `--watch` 复用同一入口

### `GET /health`

返回示例：

```json
{
  "ok": true,
  "host": "127.0.0.1",
  "port": 4317,
  "watch": true,
  "generatedAt": "2026-06-30T10:00:00.000Z"
}
```

用途：

1. 测试 server ready
2. 排查服务启动状态

### 其他路径

返回：

1. `404 Not Found`
2. 纯文本或 JSON 都可，但应固定

推荐：

1. `/` 返回 HTML
2. `/data.json`、`/health` 返回 JSON
3. 其他路径返回纯文本 `Not found`

## 服务生命周期

## 启动流程

推荐：

1. 解析 CLI 参数
2. 校验 `--watch` 是否依附 `--serve`
3. 校验 host
4. 校验 port
5. 若传入 `--output`，打印：
   - `--output only applies to static dashboard generation`
6. 先尝试构建一次初始 model
7. 若初始构建 fatal 失败则直接退出
8. 创建 server
9. `server.listen()`
10. 打印 URL
11. 若 `--open` 则打开 URL

## 运行时

服务端职责：

1. 提供 HTML
2. 提供 JSON
3. 提供健康检查

不负责：

1. 触发 agent
2. 文件系统任意 watch 回调执行命令
3. 后台任务编排

## 退出流程

必须支持：

1. Ctrl+C (`SIGINT`)
2. `SIGTERM`

推荐行为：

1. 打印 `Dashboard server stopped`
2. 关闭监听 socket
3. 干净退出 `0`

## 端口冲突处理

推荐行为：

1. `EADDRINUSE` 时直接失败
2. 输出：
   - 端口已被占用
   - 当前 host/port
   - 建议改用 `--port <n>`

不推荐：

1. 自动寻找下一个空闲端口

原因：

1. 自动改端口会让 `--open`、测试和用户预期不稳定
2. CLI 命令应尽量 deterministic

## HTML 复用策略

## 是否写入 HTML 文件

推荐结论：

1. 静态模式：写 HTML 文件
2. `--serve` 模式：不写 HTML 文件

原因：

1. 服务模式交付物是本地 URL
2. 减少隐藏副作用
3. 降低用户对“当前页面到底来自文件还是服务”的混淆

## 是否复用 `renderDashboardHtml()`

必须复用。

但建议升级签名：

```ts
renderDashboardHtml(model, {
  serveMode: boolean,
  watchEnabled: boolean,
  dataEndpoint: string,
  refreshIntervalMs: number
})
```

这样可以：

1. 静态模式关闭自动刷新脚本
2. 服务模式开启手动刷新按钮
3. `--watch` 模式开启 polling 逻辑

## 是否复用 `buildDashboardModel()`

必须复用。

原因：

1. 当前 `status/report/inspect` 聚合逻辑已经稳定
2. 继续使用同一模型能最大限度避免 V2.x 回退

## `--run + --watch` 模型策略

这是 V3.0 必须写死的规则。

推荐：

1. `buildDashboardModel(selectedRun, limit, options?)` 增加一个“selected run must persist”语义
2. 若 selected run 已在 latest N runs 内，正常返回
3. 若 selected run 不在 latest N runs 内，额外 append 或 insert 该 run
4. 该规则必须同时适用于：
   - 初始 `GET /`
   - 后续 `/data.json`
5. 这样才能保证 watch 刷新后 `selectedRunId` 不丢失

不推荐：

1. 只在首屏 include selected run
2. 刷新后回到 latest limit 默认列表

原因：

1. 这会让页面在 watch 过程中突然丢失当前选中 run
2. 用户会误以为 run 被删除或 dashboard 出错

## 浏览器端刷新策略

## 方案对比

### 方案 A：轮询 `/data.json`

实现：

1. 页面定时 `fetch('/data.json')`
2. 收到 JSON 后重绘主 UI

优点：

1. 最小实现
2. 与当前静态模型最兼容
3. 无需长连接
4. 测试最简单

缺点：

1. 存在固定轮询开销

### 方案 B：SSE

实现：

1. 服务端维护事件流
2. 页面订阅并接收更新

问题：

1. 需要额外连接状态管理
2. 测试复杂度更高
3. 当前收益不足

### 方案 C：WebSocket

不推荐。

原因：

1. 当前不需要双向通信
2. 实现和测试复杂度远高于收益

## MVP 推荐策略

V3.0 MVP 推荐：

1. `--serve` 无 `--watch`：
   - 页面显示 `Refresh data` 按钮
   - 不自动轮询
   - 点击按钮时调用 `/data.json`
   - 只局部重绘，不整页 reload
2. `--serve --watch`：
   - 页面每 `3000ms` 拉取 `/data.json`
   - 页面 `Refresh data` 按钮继续可用
   - 拉取成功后只更新前端内存模型并重绘
3. 拉取失败：
   - 页面显示轻量错误标记
   - 不清空当前已显示数据
   - 下一个周期继续重试

## 前端渲染建议

当前 `renderDashboardHtml()` 已内嵌整套前端渲染脚本。V3.0 推荐最小增量：

1. 保留 `window.__MOREAGENT_DASHBOARD_DATA__`
2. 增加 `window.__MOREAGENT_DASHBOARD_RUNTIME__`
3. 其中包含：
   - `serveMode`
   - `watchEnabled`
   - `dataEndpoint`
   - `refreshIntervalMs`

浏览器端新增 helper 建议：

1. `fetchDashboardData()`
2. `applyDashboardData(nextData)`
3. `scheduleAutoRefresh()`
4. `renderRefreshStatus()`

注意：

1. 仍然只维护一套 normalization/render 流程
2. 不要为静态模式和服务模式分叉出两套 ViewModel

## open helper 设计

推荐保留 `openInDefaultBrowser()` 名称，或在重命名后保持同等能力。

要求：

1. helper 必须同时支持 file path 和 URL
2. 静态 `dashboard --open` 传入 HTML file path
3. `dashboard --serve --open` 传入 `http://host:port/`
4. helper 不负责区分 dashboard 模式，只负责打开目标

测试要求：

1. `--open --serve` 必须断言 open target 是 URL
2. 不能把 `--serve --open` 错误实现成打开本地文件路径

## 错误语义

## 服务启动失败

应直接退出非零，典型原因：

1. host 非法
2. port 非法
3. 端口被占用
4. 初始 model 构建 fatal 失败

## `/data.json` 失败

推荐返回：

```json
{
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "..."
  }
}
```

HTTP 状态推荐：

1. `500`

页面侧行为：

1. 保留旧数据
2. 显示 refresh error

## `no runs`

返回：

1. 与 V2.2 一致的空模型
2. `/` 显示 empty dashboard
3. `/data.json` 返回 `runs: []`

## `--open` 失败语义

必须保持与 V2.3 一致：

1. 静态模式下：
   - HTML 已生成
   - open 失败 -> 提示但命令仍成功
2. 服务模式下：
   - server 已启动
   - open 失败 -> 提示但服务仍继续运行

## 安全边界

## host 限制

推荐：

1. 默认 `127.0.0.1`
2. 仅允许 `127.0.0.1` 和 `localhost`

这样可以避免：

1. 本地路径
2. merge readiness
3. worktree
4. session

这些开发态信息被非本机访问。

## 路由限制

服务端只允许：

1. `/`
2. `/data.json`
3. `/health`

不得新增：

1. 文件读取 API
2. shell API
3. merge/resume/start API

## output 与 serve 边界

必须写入文档和帮助信息：

1. `--output` 用于静态 HTML
2. `--serve` 用于本地服务
3. `--serve` 下允许传 `--output`，但必须忽略
4. `--serve` 不把 `--output` 当作 web root

## server helper 可关闭机制

推荐：

1. `startDashboardServer(options)` 返回可关闭 handle
2. handle 至少暴露：
   - `close(): Promise<void>` 或等价能力
   - `url`
   - `host`
   - `port`

这样做的原因：

1. helper/integration 测试可主动关闭 server
2. 避免端口泄漏
3. 避免测试进程悬挂

## 风险点

### 1. server 阻塞测试进程

风险：

1. 测试如果直接启动服务而不回收，容易卡住

应对：

1. server helper 返回可关闭实例
2. 测试统一 teardown

### 2. 端口冲突

风险：

1. 本机已有其他服务占用默认端口

应对：

1. deterministic fail
2. 测试中使用临时端口或先占用再验证

### 3. auto-refresh 与静态数据模型冲突

风险：

1. 页面脚本可能同时服务静态模式和服务模式

应对：

1. 只扩一层 runtime config
2. 不复制第二套渲染逻辑

### 4. 数据频繁重建性能

风险：

1. `/data.json` 每次都调用 `status/report/inspect`

应对：

1. MVP 先接受这一成本
2. polling 间隔保守设置为 3000ms
3. 未来如有必要再做短缓存

### 5. 错误 JSON 影响页面渲染

风险：

1. 某次 polling 返回错误会清空页面

应对：

1. 保留上一次成功数据
2. 只显示 refresh error

### 6. open 失败语义与 V2.3 冲突

风险：

1. 服务模式可能被误实现为 open 失败即退出

应对：

1. 明确与 V2.3 保持一致
2. open 是附加动作，不影响主能力成功

## 实施拆分

以下拆分应足够让 OpenCode 直接执行：

### Step 1: CLI 参数设计

1. 在 `cli.ts` 增加 `--serve / --watch / --host / --port`

3. 明确非法组合：
   - `--watch` without `--serve`
   - 非法 host
   - 非法 port
4. `--serve` 下若有 `--output`，打印固定提示并忽略

### Step 2: server helper

## 最终技术结论

V3.0 最合理的 MVP 实现方案是：

1. 保留静态 Dashboard 为默认模式
2. 使用 Node `http` 提供本地只读服务
3. 路由仅限 `/`、`/data.json`、`/health`
4. `GET /` 每次动态渲染最新 HTML
5. `--watch` 使用浏览器轮询而不是 SSE/WebSocket
6. `--serve` 下忽略 `--output`，且不写静态文件
7. `--run + --watch` 必须持续保留 selected run
8. open helper 同时支持 file path 和 URL
9. server helper 必须可关闭，以保证测试不会卡死

这样能在复杂度最小的前提下，把 Dashboard 从“静态快照”升级为“本地持续观察视图”。
