# MoreAgent V3.2 PRD: `moreagent init --full`

## 1. 文档目标

本文件定义 V3.2 “一键集成项目” 的产品目标与用户行为，作为 OpenCode 后续实现 `moreagent init --full` 的直接输入。

本设计基于真实源码和当前文档，不基于早期愿景假设。

已核对的真实来源：

2. `src/cli.ts`

## 2. V3.2 产品目标

### 2.1 目标

V3.2 要解决的问题不是“再多一个 profile”，而是把 MoreAgent 从“生成基础骨架”推进到“可一键接入已有项目”。

`moreagent init --full` 的目标：

1. 在已有 git 项目中一次性生成 full workflow 所需的最小完整集成材料。
2. 生成的内容不仅包含 `.moreagent/config.yaml` 和 `.opencode/agents/*.md`，还包含面向仓库维护者的接入说明文档。
3. 用户在执行完 `moreagent init --full` 后，不需要去翻 PRD 或源码，直接根据生成文档即可开始跑 full workflow。
4. 保持当前 CLI-first、本地优先、无数据库、无自动执行的产品边界。

### 2.2 非目标

V3.2 不做以下内容：

1. 不自动执行 `moreagent start`
2. 不自动探测并修改用户业务代码
3. 不自动 merge / push
4. 不接入远程服务
5. 不引入数据库
6. 不做项目模板市场
7. 不做多 runtime 支持扩展

## 3. 当前 `init` 能力分析

## 3.1 当前源码真实行为

依据 `src/commands/init.ts` 和 `src/cli.ts`，当前 `init` 已支持：

1. `moreagent init`
2. `moreagent init --profile full`

当前 `init` 的真实行为：

1. 创建 `.moreagent/`
2. 创建 `.moreagent/runs/`
3. 创建 `.moreagent/worktrees/`
4. 创建 `.moreagent/config.yaml`
5. 创建 `.moreagent/sessions.json`
6. 创建 `.moreagent/tasks.json`
7. 创建 `.moreagent/runtime-sessions.json`
8. 创建 `.opencode/agents/*.md`

当前 profile 选择逻辑：

1. 默认 profile 是 `mvp`
2. 只有 `--profile full` 会切换到 `full`
3. 当前不存在 `--full` 参数

## 3.2 当前 `init` 的限制

当前 `init` 仍然是“初始化骨架”，不是“接入现有项目”：

1. 不生成项目接入说明文档
2. 不解释 full workflow 的 9 个阶段
3. 不解释各 agent 的职责和 artifact
4. 不提示用户后续推荐命令
5. 不区分“初始化空仓库骨架”和“接入已有项目”
6. 不提供幂等级别更明确的“已存在文件如何处理”反馈

## 3.3 当前源码与文档的差异

以下差异必须在实现时明确：

1. `docs/product-requirements.current.zh-CN.md` 和 `docs/mvp-roadmap.md` 已把 `moreagent init --full` 作为 V3.2 目标，但当前源码还没有这个命令行为。
2. `README.md` 目前只展示 `moreagent init`，没有说明 `--profile full` 与未来 `--full` 的语义差异。
3. `src/cli.ts` 的 help 文案目前没有列出 `moreagent init --profile full`，更没有列出 `moreagent init --full`。
4. `src/commands/init.ts` 虽然有 `FULL_CONFIG` 和 `FULL_OPENCODE_AGENTS`，但没有生成 docs / integration guide。

结论：

1. “full workflow 已存在”是事实。
2. “一键集成 full workflow”当前还不存在。
3. V3.2 的新增重点不是 workflow 本身，而是 init 的产品化接入体验。

## 4. `init` 与 `init --full` 的差异定义

### 4.1 `moreagent init`

定位：

1. 默认初始化命令
2. 保持现有 MVP-first 行为
3. 生成最小骨架

默认结果：

1. profile = `mvp`
2. 生成 MVP 配置
3. 生成 MVP agents
4. 生成基础 state files
5. 不生成 full workflow 集成文档

### 4.2 `moreagent init --profile full`

定位：

1. 保留兼容入口
2. 生成 full profile 配置与 full agents
3. 行为上与当前源码兼容

V3.2 后建议行为：

1. 继续生成 full config 和 full agents
2. 继续创建基础 `.moreagent` 文件
3. 增加明确提示：这是 full workflow profile
4. 生成与 `--full` 相同的 full workflow 集成文档

说明：

1. 如果 `--profile full` 和 `--full` 最终生成物不一致，用户会混淆。
2. 因此 V3.2 建议把 `--profile full` 视为“full profile 初始化入口”，并与 `--full` 共享 full 生成逻辑。

### 4.3 `moreagent init --full`

定位：

1. 面向已有项目的一键 full workflow 集成入口
2. 是 V3.2 的主新增能力

核心语义：

1. 生成 full workflow 配置
2. 生成 full workflow agents
3. 生成 full workflow 接入说明文档
4. 生成项目级 next steps 指引
5. 不触发任何运行

建议用户心智：

1. `moreagent init` = 最小初始化
2. `moreagent init --full` = 一键接入完整工作流

## 5. CLI 行为定义

## 5.1 `moreagent init`

行为：

1. 默认按 `mvp` profile 初始化
2. 创建缺失的基础文件和目录
3. 生成 MVP `.opencode/agents/*.md`
4. 不生成 full workflow integration guide

成功输出建议：

1. 明确 profile = `mvp`
2. 列出创建或跳过的文件
3. 提示可升级为 full：
   `moreagent init --full`

## 5.2 `moreagent init --profile full`

行为：

1. 生成 full config
2. 生成 full agents
3. 生成基础 state files
4. 生成 full workflow integration guide

兼容性要求：

1. 这是已有参数，不能删
2. 在文案层面应说明它等价于 full workflow 初始化

## 5.3 `moreagent init --full`

行为：

1. 视为 full workflow 初始化快捷入口
2. 内部等价于 `init --profile full`，但输出文案强调“one-click project integration”
3. 生成 full profile 所需全部推荐文件
4. 生成仓库内可阅读的集成说明文档

参数关系建议：

1. `--full` 与 `--profile full` 同时出现时，允许但应归一到 full
2. `--profile mvp --full` 同时出现时，命令应报错，因为语义冲突

需要新增：

1. `src/cli.ts` 对 `--full` 的参数解析
2. 冲突参数校验
3. help 文案更新

## 5.4 重复执行时的行为

基于当前源码，`init` 已具备“只创建缺失文件、不覆盖已有文件”的弱幂等特性。

V3.2 应明确成正式产品规则：

1. 已存在目录：跳过，打印 `exists`
2. 已存在基础状态文件：跳过，打印 `exists`
3. 已存在默认 agent 文件：默认跳过，不覆盖
4. 已存在 integration guide：默认跳过，不覆盖
5. 已存在 config.yaml：默认跳过，不覆盖

重复执行输出必须清晰区分：

1. `created`
2. `skipped`
3. `already exists`
4. `manual review recommended`

## 9. 每个 agent 的职责、输入、输出 artifact

以下定义必须严格以当前实现为准，而不是抽象化重写。

## 9.1 brain

职责：

1. 分析任务
2. 输出总体规划
3. 在 PRD gate / Tech gate 阶段做 gate 判断
4. 在 repair 阶段执行 failure analysis

典型输入：

1. 用户任务
2. PRD 内容
3. 三方技术方案
4. test / review 失败上下文

输出 artifact：

1. `brain-plan.md`
2. `prd-decision.md`
3. `prd-decision-1.md`
4. `prd-decision-2.md`
5. `tech-review.md`
6. `tech-review-1.md`
7. `tech-review-2.md`
8. `failure-analysis.md`

## 9.2 product

职责：

1. 编写 PRD
2. 在 PRD gate 失败后修订 PRD
3. 在 product owner repair 场景下回修需求

典型输入：

1. 用户任务
2. brain 规划
3. PRD review 反馈
4. failure-analysis 指定为 `Owner: product`

输出 artifact：

1. `prd.md`
2. `prd-revision-1.md`
3. `prd-revision-2.md`
4. `prd-revision-repair-<n>.md`

## 9.3 frontend

职责：

1. 评审 PRD
2. 编写 frontend plan
3. 实施前端代码改动
4. 在 repair 中承担 frontend owner 修复

典型输入：

1. PRD
2. gate 反馈
3. failure-analysis 结果
4. 同轮其它相关 artifact

输出 artifact：

1. `frontend-prd-review.md`
2. `frontend-plan.md`
3. `frontend-plan-revision-<n>.md`
4. `frontend-implementation.md`
5. `frontend-repair-<n>.md`
6. `frontend-implementation-repair-<n>.md`

## 9.4 backend

职责：

1. 评审 PRD
2. 编写 backend plan
3. 实施后端代码改动
4. 在 repair 中承担 backend owner 修复

输出 artifact：

1. `backend-prd-review.md`
2. `backend-plan.md`
3. `backend-plan-revision-<n>.md`
4. `backend-implementation.md`
5. `backend-repair-<n>.md`
6. `backend-implementation-repair-<n>.md`

## 9.5 tester

职责：

1. 评审 PRD 的 testability
2. 编写 test plan
3. 执行测试
4. 在 tester owner repair 场景下回修测试

协议要求：

1. 测试报告顶部必须有 `Result: PASS` 或 `Result: FAIL`

输出 artifact：

1. `test-prd-review.md`
2. `test-plan.md`
3. `test-plan-revision-<n>.md`
4. `test-report.md`
5. `test-report-retry-<n>.md`
6. `test-repair-<n>.md`
7. `test-plan-repair-<n>.md`

## 9.6 reviewer

职责：

1. 对最终实现做代码审查
2. 以 artifact decision 驱动是否进入 repair

协议要求：

1. review 报告顶部必须有 `Decision: APPROVED` 或 `Decision: CHANGES_REQUESTED`

输出 artifact：

1. `review-report.md`
2. 相关 repair 回路中的 `review-report` 派生文件

## 11. 幂等策略

V3.2 必须是安全幂等的。

### 11.1 文件已存在时

默认规则：

1. 不覆盖用户已存在文件
2. 不删除已有文件
3. 不重写用户自定义 agent prompt

### 11.2 覆盖策略

本版本建议：

1. 不新增 `--force`
2. 不自动覆盖
3. 使用 skip + prompt 方式

原因：

1. 这是最安全的 MVP 设计
2. 能避免误伤用户定制 agent / config
3. 覆盖策略可以留给后续版本

### 11.3 提示策略

需要明确打印：

1. 哪些文件是 `created`
2. 哪些文件是 `skipped`
3. 哪些文件建议人工检查

## 12. 安全边界

V3.2 必须遵守以下边界：

1. 不删除用户文件
2. 不覆盖用户自定义 agent
3. 不自动执行 `start`
4. 不自动 merge
5. 不自动修改业务源码
6. 不依赖数据库
7. 不要求联网

## 13. 验收标准

V3.2 达标标准：

1. `moreagent init --full` 可在已有 git 仓库中成功执行。
2. 会生成 full workflow 所需 config、state files、agents、integration guide。
3. 已存在文件默认不覆盖。
4. 重复执行不会破坏已有用户定制内容。
5. `README` 和 CLI help 能解释 `init`、`init --profile full`、`init --full` 的差异。
6. 用户按生成的 integration guide 能完成一次 full workflow 试跑准备。

## 14. 最终产品结论

V3.2 不是再造一个 workflow，而是把已有 full workflow 做成“可一键接入已有项目”的产品入口。

这意味着实现重点应放在：

1. 明确的 CLI 语义
2. 可读的生成结果
3. 安全的幂等策略
4. 可直接执行的接入说明

而不是继续扩展运行时复杂度。
