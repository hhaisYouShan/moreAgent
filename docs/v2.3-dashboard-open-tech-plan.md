# V2.3 Dashboard `--open` Technical Plan

## 目标

在现有 static dashboard 流程上追加一个很小的交付动作：

- `moreagent dashboard --open`

实现目标不是引入新模式，而是：

1. 先生成 HTML
2. 再调用系统默认浏览器打开该文件

## 当前实现分析

根据当前 `src/commands/dashboard.ts`，现有链路是：

1. `dashboardCommand(options)`
2. `buildDashboardModel(options.run, limit)`
3. `renderDashboardHtml(model)`
4. `writeDashboardHtml(outputPath, html)`
5. `console.log("Dashboard written to ...")`

当前结论：

- 数据采集已稳定
- HTML 渲染已稳定
- 输出路径逻辑已稳定
- V2.3 只需要在写文件成功后追加一个 open 步骤

## 改动范围

V2.3 只建议改：

1. `src/commands/dashboard.ts`
2. `src/cli.ts`
3. `test/regression.test.js`

## 不改范围

## 命令参数设计

建议在 `DashboardOptions` 中新增：

```ts
export interface DashboardOptions {
  run?: string;
  limit?: number;
  output?: string;
  open?: boolean;
}
```

CLI 建议支持：

```bash
moreagent dashboard --open
```

并允许与以下参数自由组合：

1. `--run`
2. `--limit`
3. `--output`

## 技术决策

## 决策 1：`--open` 不是新命令，而是 `dashboard` 的附加 flag

原因：

1. 与 `--run / --limit / --output` 组合自然
2. 语义清晰：仍然是 dashboard，只是生成后顺手打开
3. 不需要新增第二个命令入口

## 决策 2：open 行为必须在 write 成功之后

推荐流程：

```ts
function dashboardCommand(options) {
  const model = buildDashboardModel(...);
  if (!model) fail;

  const html = renderDashboardHtml(model);
  writeDashboardHtml(outputPath, html);
  console.log(`Dashboard written to ${outputPath}`);

  if (options.open) {
    tryOpenDashboard(outputPath);
  }
}
```

原因：

1. open 的对象必须是已存在的 HTML 文件
2. 若 write 失败，不应该做无效 open

## 决策 3：open 失败不应抹掉生成成功

推荐行为：

- HTML 已写成功后，open 失败只输出 warning
- 命令建议保持成功退出

原因：

1. 主任务是生成 dashboard
2. `--open` 只是附加体验
3. 浏览器关联故障不应被误判成 dashboard 生成失败

## 平台兼容策略

V2.3 不需要做跨平台完美抽象，但需要有明确策略。

建议：

- macOS：`open <file>`
- Linux：`xdg-open <file>`
- Windows：使用系统等价打开方式

技术上应封装为单个 helper：

```ts
function openInDefaultBrowser(filePath: string): { ok: true } | { ok: false; message: string }
```

helper 内部负责根据平台选择命令。

## 推荐新增 helper

## 1. `resolveDashboardOutputPath()`

### 输入

```ts
resolveDashboardOutputPath(output?: string): string
```

### 输出

- 若用户传了 `--output`，返回该路径
- 否则返回默认 `.moreagent/dashboard/index.html`

### 目的

确保：

- write 和 open 使用完全同一条路径

## 2. `openInDefaultBrowser(filePath)`

### 输入

```ts
openInDefaultBrowser(filePath: string): { ok: true } | { ok: false; message: string }
```

### 输出

- 成功：`{ ok: true }`
- 失败：`{ ok: false, message }`

### 规则

1. 只在文件已成功写入后调用
2. 只负责 open，不负责生成 HTML
3. 不抛未处理异常到最外层

## 3. `printOpenResult(filePath, result)`

### 输入

```ts
printOpenResult(filePath: string, result: { ok: true } | { ok: false; message: string }): void
```

### 输出

- 成功时输出：
  - `Opened dashboard in default browser`
- 失败时输出：
  - `Open failed: ...`
  - `Dashboard was still generated at ...`

## 数据流

V2.3 的数据流不变，只在尾部追加 open 步骤。

## 不带 `--open`

```text
build model -> render html -> write file -> print path
```

## 带 `--open`

```text
build model -> render html -> write file -> print path -> try open -> print open result
```

## 错误处理规则

## 1. model 构建失败

例如：

- fatal status error
- selected run 不在 prefetched range

处理：

- 命令失败
- 不写文件
- 不尝试 open

## 2. HTML 渲染失败

处理：

- 命令失败
- 不尝试 open

## 3. 文件写入失败

处理：

- 命令失败
- 不尝试 open

## 4. open 失败

处理：

- 保留已写文件
- 打印 warning
- 不回滚已生成文件
- 推荐 exit `0`

## 为什么推荐 open 失败仍 exit 0

这是 V2.3 的关键判断。

推荐理由：

1. `dashboard` 主任务已经完成
2. 用户真正关心的 HTML 产物已经落地
3. 若 exit 非 0，脚本调用者可能误判 dashboard 生成失败

如果未来需要更严格模式，可以另做扩展，但 V2.3 不建议引入。

## 与现有参数组合规则

## 1. `--run + --open`

- 先按 `--run` 生成
- 再打开生成结果

## 2. `--limit + --open`

- 先按 `--limit` 生成
- 再打开生成结果

## 3. `--output + --open`

- 先写 `--output`
- 再打开该路径

## 4. 多 flag 同时存在

优先级：

1. `--run / --limit / --output` 决定生成内容与输出路径
2. `--open` 只决定是否在生成后尝试打开

## 风险点

### 1. 平台差异

不同平台默认打开文件的命令不同。

V2.3 风险不在于数据逻辑，而在于平台命令封装。

### 2. 测试环境无法真实打开浏览器

因此测试不能依赖真实浏览器弹起，建议：

- mock open helper
- 断言调用与返回值

### 3. open 失败时 exit code 语义

若实现者误把 open 失败当成整体失败，会破坏“HTML 已成功生成”的语义。

必须明确：

- open failure ≠ dashboard generation failure
