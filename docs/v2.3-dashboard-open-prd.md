# V2.3 Dashboard `--open` PRD

## 背景

到当前阶段，`moreagent` 的 Dashboard 路线已经演进为：

1. V2.0：static dashboard
2. V2.0.1：dashboard hardening
3. V2.1：dashboard usability enhancement
4. V2.2：dashboard resilience

当前 Dashboard 的核心使用方式仍然是：

1. 运行 `moreagent dashboard`
2. 生成 `.moreagent/dashboard/index.html`
3. 用户手动去文件系统里打开该 HTML

这条链路已经稳定，但还存在一个明显的体验摩擦：

- 用户拿到输出路径后，还需要手动打开浏览器或 Finder

V2.3 的目标是补一个很小但高频有用的能力：

- `moreagent dashboard --open`

它不是新架构，不是服务化，也不是刷新机制，只是把“生成 HTML 后再打开”这一步补上。

## 目标

新增：

```bash
moreagent dashboard --open
```

行为要求：

1. 先生成 Dashboard HTML
2. 再使用系统默认浏览器打开该 HTML 文件
3. 支持与 `--run / --limit / --output` 组合
4. 若打开失败，不影响 “HTML 已生成成功” 这一事实
5. 打开失败时必须给出明确提示

## 用户痛点

### 1. 当前使用链路多一步手工操作

用户现在通常要：

1. 运行命令
2. 复制路径
3. 手动双击或 `open`

这一步不复杂，但高频、重复、纯机械。

### 2. `--output` 场景更容易断链

当用户指定了自定义输出路径，例如：

```bash
moreagent dashboard --output /tmp/moreagent-dashboard.html
```

命令虽然成功了，但用户还要再去定位并打开该文件。

### 3. 当前 static dashboard 已经够用，不需要直接上 server

V2.3 不需要引入 `--serve`、`--watch`、自动刷新，只需要把现有静态产物的最后一步交付体验补齐。

## 产品目标

V2.3 只解决一件事：

- **生成完 HTML 后可直接打开**

这意味着：

1. 不改变 dashboard 数据模型
2. 不改变 static HTML 形态
3. 不改变现有 `--run / --limit / --output` 语义
4. 不引入新的运行模式

## 非目标

V2.3 不做：

1. `dashboard --serve`
2. `dashboard --watch`
3. 自动刷新
4. 执行按钮
5. 登录
6. 浏览器端热更新
7. 多窗口管理
8. 打开后监控文件变更

## 命令设计

## 基本命令

```bash
moreagent dashboard --open
```

含义：

1. 先生成 dashboard HTML
2. 再用系统默认浏览器打开该文件

## 与已有参数组合

### 1. 与 `--run` 组合

```bash
moreagent dashboard --run run-2026-06-30-abc123 --open
```

语义：

- 先按该 run 作为默认选中 run 生成 dashboard
- 再打开生成结果

### 2. 与 `--limit` 组合

```bash
moreagent dashboard --limit 20 --open
```

语义：

- 先按 limit=20 生成 dashboard
- 再打开生成结果

### 3. 与 `--output` 组合

```bash
moreagent dashboard --output /tmp/moreagent-dashboard.html --open
```

语义：

- 先写到指定输出路径
- 再打开这个指定路径

### 4. 多参数组合

```bash
moreagent dashboard --run run-xxx --limit 5 --output /tmp/dash.html --open
```

语义：

- 完全遵循现有参数行为
- `--open` 只是在成功生成后追加一步“打开输出文件”

## 核心行为规则

## 1. 生成优先，打开其次

顺序必须固定：

1. build dashboard model
2. render HTML
3. write HTML file
4. try open

绝对不能：

1. 先尝试 open
2. 再写文件

## 2. 打开失败不回滚生成成功

如果：

- HTML 已经成功写入
- 但系统默认浏览器打开失败

则应视为：

- dashboard 生成成功
- open 动作失败

命令必须明确提示：

- `Dashboard written to ...`
- `Open failed: ...`

## 3. 打开失败的推荐行为

推荐行为：

- 保持命令整体为成功退出
- 但输出明确 warning

原因：

1. 主要任务“生成 HTML”已经成功
2. `--open` 是增强体验，不是 HTML 生成的必要条件
3. 若因浏览器环境问题让整个命令失败，会误导用户认为 dashboard 没生成

## 4. 打开路径必须等于最终输出路径

如果指定了：

- 默认输出路径：`.moreagent/dashboard/index.html`
- 或自定义 `--output`

则 `--open` 必须打开 **最终实际写入的那个路径**。

## 5. 不改变已有 CLI 语义

V2.3 不能改变：

1. `dashboard --run`
2. `dashboard --limit`
3. `dashboard --output`

已有行为。

`--open` 只能追加，不应重写原有逻辑。

## 成功路径

## 场景 1：默认生成并打开

```bash
moreagent dashboard --open
```

预期：

1. 生成 `.moreagent/dashboard/index.html`
2. 输出生成成功路径
3. 系统默认浏览器打开该 HTML

## 场景 2：指定 run 并打开

```bash
moreagent dashboard --run run-xxx --open
```

预期：

1. 用该 run 生成 dashboard
2. 输出路径
3. 浏览器打开

## 场景 3：指定 output 并打开

```bash
moreagent dashboard --output /tmp/dash.html --open
```

预期：

1. 写 `/tmp/dash.html`
2. 浏览器打开 `/tmp/dash.html`

## 失败路径

## 1. dashboard 生成失败

例如：

- `status --json` fatal
- output 写入失败

预期：

- 命令失败
- 不尝试 open

## 2. dashboard 生成成功，但打开失败

例如：

- 系统 `open` 命令不可用
- 平台命令失败
- 浏览器关联异常

预期：

1. HTML 仍然保留
2. 输出成功生成路径
3. 输出 `Open failed`
4. 不误导用户说 dashboard 生成失败

## 文案建议

成功生成：

- `Dashboard written to /path/to/file.html`

打开成功：

- `Opened dashboard in default browser`

打开失败：

- `Open failed: <reason>`
- `Dashboard was still generated at /path/to/file.html`

## 验收标准

1. `moreagent dashboard --open` 可用
2. 仍默认生成 `.moreagent/dashboard/index.html`
3. `--open` 可与 `--run` 组合
4. `--open` 可与 `--limit` 组合
5. `--open` 可与 `--output` 组合
6. 打开动作发生在 HTML 写入之后
7. 生成失败时不尝试 open
8. open 失败时 HTML 文件仍保留
9. open 失败时有明确提示
10. open 失败不掩盖“HTML 已生成”事实
11. 不引入 server / watch / 自动刷新
12. 不改变现有 dashboard JSON / HTML 生成逻辑
