# V3.0 Dashboard `--serve / --watch / auto-refresh` PRD

## 背景

截至 V2.4，`moreagent dashboard` 已具备以下能力：

1. 生成静态 HTML Dashboard
2. 支持 `--run / --limit / --output`
3. 支持 `--open`
4. 支持 empty state、MVP run 降级、JSON/debug、可读性增强与 resilience

当前问题不是“有没有 Dashboard”，而是“Dashboard 是否足够适合持续观察”。

静态 HTML 的优点已经验证成立：

1. 无服务端依赖
2. 单文件可归档
3. 实现与测试成本低
4. 对本地 CLI 工具非常友好

但静态模式在以下场景存在明显摩擦：

1. `moreagent start --loop` 运行时，用户想持续看 runs 变化
2. repair/gate 状态变化后，用户需要反复重新生成页面
3. 队列执行或长时运行时，用户想保留一个浏览器页签观察状态

因此 V3.0 不是替换静态 Dashboard，而是在其上增加本地服务模式：

1. `moreagent dashboard --serve`
2. `moreagent dashboard --serve --watch`
3. auto-refresh

核心目标是把“手动反复重新生成 HTML”升级成“本地只读观察页面”，但不引入远程服务化复杂度。

## 路线定位

V3.0 是 Dashboard 路线的下一阶段，不回退 V2.x 的静态 HTML 路线：

1. V2.0 到 V2.4：静态 HTML 可用、可读、可诊断
2. V3.0：本地 serve/watch/auto-refresh

静态 Dashboard 必须保留，原因是：

1. 归档和分享仍然需要单文件 HTML
2. 故障排查时静态文件更容易复现
3. `--serve` 不应成为唯一入口

## 产品目标

V3.0 目标是新增一个本地只读观察模式：

1. `moreagent dashboard --serve`
2. `moreagent dashboard --serve --watch`
3. `moreagent dashboard --serve --open`

它应满足：

1. 保留现有静态 Dashboard 行为
2. 支持本地浏览器访问同样的 Dashboard UI
3. 在 watch 模式下自动刷新数据
4. 不要求用户手工重新运行 `moreagent dashboard`
5. 不暴露执行型按钮，不承担工作流控制职责

## 用户问题

### 1. 静态页面适合查看，不适合持续观察

当前用户通常要：

1. 运行 `moreagent dashboard`
2. 打开 HTML
3. agent 状态变化后再重新生成
4. 浏览器重新打开或手工刷新静态文件

这对单次验收可接受，但对长时观察不够顺滑。

### 2. 浏览器中的页面与 CLI 当前状态容易脱节

当前静态文件只是生成时刻的快照。只要 runs 状态变化：

1. repair session 增加
2. overallStatus 变化
3. gate/test/review 结果变化
4. merge readiness 变化

浏览器里看到的就是旧数据。

### 3. 用户需要“本地观察页”，而不是“服务化平台”

当前更实际的需求是：

1. 本地起一个只读服务
2. 浏览器持续查看
3. 页面自动更新

而不是：

1. 远程部署
2. 登录权限
3. 多人协作
4. 复杂前后端分离系统

## 产品范围

V3.0 范围内必须覆盖：

### 1. `moreagent dashboard`

继续保留现有行为：

1. 生成静态 HTML
2. 默认输出 `.moreagent/dashboard/index.html`
3. 不启动服务
4. 不自动刷新

### 2. `moreagent dashboard --open`

继续保留现有行为：

1. 先生成静态 HTML
2. 再打开文件
3. 若 open 失败，HTML 仍视为已生成

### 3. `moreagent dashboard --serve`

新增行为：

1. 启动本地 Dashboard 服务
2. 默认监听 `127.0.0.1`
3. 默认端口使用固定值
4. 浏览器访问一个本地 URL
5. 默认返回与静态 Dashboard 接近的同构 UI
6. 进程保持运行，直到用户 Ctrl+C
7. 不写静态 HTML 文件
8. 若同时传入 `--output`，CLI 打印：
   - `--output only applies to static dashboard generation`

### 4. `moreagent dashboard --serve --open`

新增行为：

1. 启动本地服务
2. 成功监听后自动打开服务 URL
3. open 失败不影响服务继续运行
4. 用户仍可手动访问打印出的 URL

### 5. `moreagent dashboard --serve --watch`

新增行为：

1. 启动本地服务
2. 开启自动刷新数据能力
3. 页面无需手工刷新即可看到更新
4. 不要求真正的文件系统 watch；MVP 可用 polling 驱动
5. `--watch` 才启用 `3000ms` polling `/data.json`

### 6. 参数兼容

V3.0 必须明确 `--serve` 模式下是否继续支持：

1. `--run`
2. `--limit`
3. `--output`

本 PRD 推荐：

1. `--run` 支持
2. `--limit` 支持
3. `--output` 允许传入，但在 `--serve` 下被忽略

## 非目标

V3.0 明确不做：

1. 远程服务
2. 登录
3. 多人协作
4. merge/resume/start 等执行型按钮
5. 云端部署
6. 任意文件浏览
7. 复杂 WebSocket 架构
8. 独立前端工程
9. React/Vue 重构
10. dashboard 成为新的 workflow 执行入口

## CLI 行为定义

### `moreagent dashboard`

行为：

1. 生成静态 HTML
2. 写入默认或指定输出路径
3. 输出文件路径
4. 退出

### `moreagent dashboard --open`

行为：

1. 生成静态 HTML
2. 尝试打开文件
3. open 失败时提示 `Open failed`
4. 退出

### `moreagent dashboard --serve`

行为：

1. 启动本地 HTTP 服务
2. 不写静态 HTML 文件
3. 若同时传入 `--output`，CLI 打印：
   - `--output only applies to static dashboard generation`
4. 控制台打印服务 URL
5. 保持运行直到 Ctrl+C
6. `GET /` 每次请求都动态渲染当前最新 HTML
7. 默认不自动轮询
8. 页面提供 `Refresh data` 按钮

### `moreagent dashboard --serve --open`

行为：

1. 启动服务
2. 服务成功监听后自动打开 URL
3. open 失败只提示，不退出服务
4. 打开的目标是 `http://host:port/`，不是文件路径

### `moreagent dashboard --serve --watch`

行为：

1. 启动服务
2. 页面通过客户端自动刷新机制获取最新数据
3. `GET /` 每次请求仍动态渲染当前最新 HTML
4. 页面 `Refresh data` 按钮继续可用
5. 仅 `--watch` 启用 `3000ms` polling `/data.json`
6. watch 语义是“服务端允许数据持续更新”，不是“CLI 自动执行新的 runs”

### `moreagent dashboard --serve --port 4317`

行为：

1. 监听用户指定端口
2. 若端口被占用，命令失败
3. 不静默切换到其他端口

### `moreagent dashboard --serve --host 127.0.0.1`

行为：

1. 监听指定 host
2. MVP 默认仍建议仅支持 loopback 地址
3. 若用户传入非 loopback 地址，MVP 可拒绝并提示安全边界

### `moreagent dashboard --serve --limit 20`

行为：

1. 服务端数据预取最近 20 个 runs
2. 页面侧边栏最多展示这 20 个 runs
3. auto-refresh 时仍只更新这 20 个范围内的数据模型
4. 若同时指定了 `--run <id>`，该 run 必须额外保留在模型中，即使它不在 latest 20 里

### `moreagent dashboard --serve --run <id>`

行为：

1. 启动时预选中该 run
2. 即使它不在 latest `--limit` 范围内，也必须额外 include 到 DashboardModel
3. 后续 watch 刷新时也必须持续保留该 selected run
4. 避免 `selectedRunId` 因后续 runs 排序变化而丢失

## 服务端产品设计

## 数据视图

V3.0 页面继续消费 DashboardModel 衍生数据，不新增第二套 schema。

页面至少保留以下区域：

1. Run List
2. Enhanced Summary
3. Workflow Progress
4. Gate / Test / Review
5. Merge Readiness
6. Repair Sessions
7. Sessions
8. JSON / Debug

这意味着 V3.0 的变化重点是“交付与刷新方式”，不是重新设计页面信息结构。

## 页面入口

推荐页面与接口：

1. `GET /`
2. `GET /data.json`
3. `GET /health`

### `GET /`

返回：

1. HTML shell
2. 内嵌当前最新数据
3. 浏览器端刷新脚本

要求：

1. 每次请求都动态渲染当前最新 HTML
2. 不使用服务启动时的一次性 HTML 快照

### `GET /data.json`

返回：

1. 与当前 DashboardModel 等价的 JSON 数据
2. 作为 `Refresh data` 和 `--watch` 的统一刷新来源

### `GET /health`

返回：

1. `{ ok: true }`
2. 可附带 host/port/watch/generatedAt 等诊断字段

## 是否保留静态 Dashboard

必须保留。

V3.0 推荐规则：

1. 默认 `dashboard` 仍生成静态 HTML
2. `dashboard --serve` 启动服务
3. 两者可以共存
4. `--serve` 不替代静态模式

## 是否自动刷新

推荐：

1. 仅在 `--watch` 时自动刷新
2. 仅 `--serve` 但未加 `--watch` 时，页面提供手动刷新按钮
3. 这样能清晰区分“服务模式”和“持续更新模式”

原因：

1. `--serve` 先解决“本地 URL 观察”
2. `--watch` 再开启“数据自动变动”
3. 用户可明确控制性能和行为

## 是否需要浏览器端手动刷新按钮

需要。

原因：

1. `--serve` 无 `--watch` 时仍需要更新入口
2. `--watch` 失败或 polling 异常时仍有回退路径
3. 更适合 debug 与验收

按钮语义推荐：

1. `Refresh data`
2. 调用 `/data.json`
3. 只局部重绘，不重新加载整页
4. `--serve` 和 `--serve --watch` 都保留这个按钮

## 刷新策略比较

V3.0 必须比较以下三种方案：

### 方案 A：浏览器轮询 `/data.json`

优点：

1. 不新增依赖
2. 基于现有 Node `http` 即可实现
3. 最容易测试
4. 与静态 Dashboard 模型最兼容
5. 调试成本低

缺点：

1. 有固定间隔延迟
2. 数据可能重复拉取

### 方案 B：SSE

优点：

1. 单向推送语义比轮询更自然
2. 比 WebSocket 简单

缺点：

1. 需要更复杂的连接生命周期管理
2. 仍要处理重连、事件格式、服务关闭
3. 对当前本地 CLI MVP 来说收益不足

### 方案 C：WebSocket

优点：

1. 最强实时性

缺点：

1. 复杂度最高
2. 测试成本最高
3. 明显超出当前 moreAgent 本地工具的必要复杂度

### MVP 推荐

V3.0 MVP 推荐：

1. `--serve` 提供本地 HTTP 服务
2. `--serve` 默认不自动轮询
3. 页面提供 `Refresh data`
4. `--watch` 使用浏览器轮询 `/data.json`
5. 先不做 SSE
6. 不做 WebSocket

## 安全边界

V3.0 必须明确以下边界：

### 1. 默认只监听 `127.0.0.1`

原因：

1. Dashboard 展示本地工程路径、worktree、错误输出
2. 不适合暴露到局域网
3. 本功能定位就是本地开发观察页

### 2. 不开放远程访问

MVP 推荐：

1. 默认 host 固定 `127.0.0.1`
2. 即使暴露 `--host`，也只接受 `127.0.0.1` 或 `localhost`
3. 若传 `0.0.0.0`，直接拒绝

### 3. 不提供执行型按钮

页面不能提供：

1. merge
2. resume
3. start
4. clean

原因：

1. 这些都带副作用
2. 当前 Dashboard 仍应保持只读

### 4. 不执行用户输入命令

浏览器端不接受任意 CLI 输入，也不构造 shell。

### 5. 不暴露任意文件读取

服务端仅提供固定路由：

1. `/`
2. `/data.json`
3. `/health`

不能出现：

1. `?path=...`
2. `/file/...`
3. 任意 artifact 文件直读

### 6. `output path` 与 `serve path` 分离

需要明确：

1. `--output` 是静态文件输出路径
2. `--serve` 是 HTTP 服务 URL
3. `--serve` 下允许传 `--output`，但必须忽略
4. 不能把 `--output` 当作静态目录映射根

## 错误处理与状态语义

### 1. 数据构建失败

如果构建 DashboardModel 失败：

1. `/data.json` 返回错误 JSON
2. 页面显示可读错误状态
3. 服务本身仍可继续运行

例外：

1. 启动首帧数据都无法构建，且服务模式无法提供任何有效页面时，命令应直接失败

### 2. no runs

行为：

1. 服务可正常启动
2. `/` 显示 empty dashboard
3. `/data.json` 返回空 runs 模型

### 3. Open failed

若 `--serve --open` 下浏览器打开失败：

1. 服务继续运行
2. 打印 `Open failed`
3. 打印本地 URL

说明：

1. open helper 必须同时支持 file path 和 URL
2. 静态 `--open` 打开 HTML file path
3. `--serve --open` 打开 `http://host:port/`

### 4. port conflict

若端口占用：

1. 命令失败
2. 清晰提示端口已被占用
3. 建议用户换端口

## 验收标准

V3.0 产品层验收至少包括：

1. `moreagent dashboard` 继续生成静态 HTML
2. `moreagent dashboard --open` 行为不回退
3. `moreagent dashboard --serve` 可在本地打开 Dashboard
4. `moreagent dashboard --serve --open` 自动打开 URL
5. `moreagent dashboard --serve --watch` 页面可自动更新数据
6. `--run / --limit` 在服务模式下仍生效
7. `--run + --watch` 下 selected run 不会因刷新掉出模型
8. no runs 仍显示 empty dashboard
9. MVP run 仍显示 workflow unavailable
10. 页面不新增任何执行型按钮
11. 服务默认仅监听 `127.0.0.1`
12. `--serve --output` 不写静态 HTML，且打印固定提示
13. `--serve --open` 打开的是 URL，不是文件路径

## 测试可关闭性要求

V3.0 文档层必须明确：

1. `startDashboardServer` 应返回可关闭 handle
2. helper 级测试必须可以直接调用 `close()`
3. CLI 常驻测试必须使用 `spawn + kill + wait exit`
4. 不能让 `npm test` 因 server 常驻而卡死

## 用户命令示例

### 静态模式

```bash
moreagent dashboard
moreagent dashboard --open
moreagent dashboard --run run-2026-06-30T12-00-00-abc123 --limit 10 --open
```

### 服务模式

```bash
moreagent dashboard --serve
moreagent dashboard --serve --open
moreagent dashboard --serve --watch
moreagent dashboard --serve --port 4317
moreagent dashboard --serve --host 127.0.0.1
moreagent dashboard --serve --limit 20
moreagent dashboard --serve --run run-2026-06-30T12-00-00-abc123
```

## 对 OpenCode 实现者的产品结论

V3.0 的正确实现方向是：

1. 保留静态 Dashboard 为默认模式
2. `--serve` 增加本地只读服务模式
3. `--watch` 只解决数据刷新，不引入工作流执行
4. refresh MVP 采用浏览器轮询 `/data.json`
5. `GET /` 每次都动态渲染最新 HTML
6. 安全边界固定在 localhost 和只读固定路由

这不是一个“Web 平台”，而是一个“本地 CLI 的可持续观察视图”。
