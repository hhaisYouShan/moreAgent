# V2.1 Dashboard 可读性增强 PRD

## 背景

`moreagent` 在 V2.0 已完成静态 HTML Dashboard：

- runs 列表
- run detail
- workflow report
- gate / test / review
- repair sessions
- merge readiness
- sessions
- JSON / debug

V2.0.1 已完成 dashboard hardening，并已通过：

- `npm test`: 58 passed
- `npm run build`: 无错误

当前 Dashboard 在功能覆盖上已经够用，但主要问题已经从“有没有信息”转向“信息是否一眼能读懂”。

## 用户痛点

### 1. Run 列表扫读成本仍然偏高

当前 sidebar 虽然已有 badge 和基础信息，但用户仍需要停下来辨认：

- `failed`
- `running`
- `merge ready`
- `mvp/full`
- recommendation

尤其当最近 runs 较多时，失败 run 和 merge-ready run 还不够跳眼。

### 2. 顶部 Summary 仍然不够像“验收结论区”

当前页面虽然有 Header 和 decision 信息，但用户判断一次 run 的核心结论时，还需要在多个区块来回看：

- overallStatus
- recommendation
- canResume
- canMerge
- mainClean
- worktree exists

这部分应收敛成一个真正的“当前 run 总结区”。

### 3. Workflow Progress 的状态层次还不够明确

full workflow 的 9 phases 虽然能显示，但还不够清楚地区分：

- completed
- failed
- pending

而 MVP run 也需要更清晰的降级展示，避免用户误以为 workflow 数据丢失。

### 4. Gate / Test / Review 的状态虽然齐全，但视觉优先级还不够

当前用户最关心的四个判断是：

- PRD Gate
- Tech Gate
- Test
- Review

但它们需要更强的一眼识别能力，尤其是：

- `APPROVED`
- `CHANGES_REQUESTED`
- `PASS`
- `FAIL`
- `unknown`

### 5. Merge Readiness 缺少“为什么”

仅显示 `MERGE_READY` 或 `BLOCKED` 不够。

用户还需要快速知道：

- 为什么 ready
- 为什么 blocked
- 是主项目 dirty
- 还是 worktree 丢失
- 还是 `canMerge = false`

### 6. JSON / Debug 对普通用户的干扰仍然偏强

JSON/debug 对诊断很有用，但不是大多数用户打开 Dashboard 的第一目标。

需要让它：

- 始终可访问
- 但默认不抢主内容阅读优先级

## V2.1 目标

V2.1 只做“小而实用”的可读性增强，不改变 Dashboard 架构，不新增 server，不改变 JSON schema。

目标是让用户打开 Dashboard 后，在 10 到 15 秒内回答：

1. 当前 run 是成功、失败还是运行中
2. 是否 merge ready
3. 如果 blocked，为什么 blocked
4. 如果 failed，失败集中在哪个层面
5. full workflow 当前推进到了哪一步
6. 如果是 MVP run，为什么没有 workflow 图

## 非目标

V2.1 不做：

1. 本地 server
2. `--serve`
3. `--watch`
4. 自动刷新
5. 执行型按钮，如 merge / resume / start
6. 登录
7. React / Vue
8. 外部依赖
9. 复杂图表
10. JSON schema 变更
11. CLI 行为变更

## 页面结构

V2.1 维持 V2.0 的页面大结构，但增强信息分层：

1. Run 列表 Sidebar
2. Enhanced Summary 顶部结论区
3. Workflow Progress
4. Gate / Test / Review 卡片区
5. Merge Readiness 解释区
6. Repair Sessions
7. Sessions 表格
8. JSON / Debug

## 1. Run 列表可读性增强

### 展示目标

用户在 sidebar 一眼分辨：

- failed
- running
- merge ready
- full / mvp
- task 是什么
- createdAt 大概何时

### 展示字段

每个 run item 建议展示：

- run id
- task 摘要
- status
- recommendation
- profile
- createdAt

### 视觉规则

- `FAILED`
  - 强红色左边界或状态点
  - recommendation badge 同时高亮
- `RUNNING`
  - 黄色或橙色状态点
- `MERGE_READY`
  - 绿色 recommendation badge
- `full`
  - 明确用 `FULL` profile badge
- `mvp`
  - 明确用 `MVP` profile badge

### task 过长展示规则

- sidebar 内 task 只显示单行摘要
- 超长时截断并显示省略号
- 完整 task 在：
  - hover title
  - 或右侧 Summary 区完整展示

不建议 sidebar 内多行展开 task，否则 runs 列表密度会明显下降。

### createdAt 展示规则

- sidebar 显示短格式时间
- run detail 里保留完整 ISO 时间

## 2. 顶部 Summary 区增强

### 目标

顶部 Summary 变成 Dashboard 的“核心结论区”。

### 必须展示字段

- run id
- task
- status
- overallStatus
- recommendation
- canResume
- canMerge
- mainClean
- worktree exists

### 推荐结构

分成两层：

#### 第一层：主结论

- run id
- task
- overallStatus
- recommendation

#### 第二层：操作条件

- canResume
- canMerge
- mainClean
- worktree exists

### 视觉规则

- overallStatus 是最大信息层级
- recommendation 是第二层级
- canResume / canMerge / mainClean / worktree exists 用较小状态 pill

### 典型文案

- `PASSED`
- `FAILED`
- `RUNNING`
- `PARTIAL`

以及：

- `MERGE_READY`
- `BLOCKED`
- `NEEDS_REPAIR`
- `NEEDS_REVIEW`

## 3. Workflow Progress 增强

### full workflow 展示范围

full workflow 固定展示 9 phases：

1. brain
2. prd
3. prd-review
4. prd-gate
5. tech-plan
6. tech-gate
7. implementation
8. test
9. review

### 状态规则

- completed
  - 绿色
- failed
  - 红色
- pending
  - 灰色

### 展示原则

- phase 顺序固定
- 不因某些 phase 未开始而隐藏
- 失败 phase 明确标出
- completed phases 连接起来时应有明显“推进感”

### MVP run 降级规则

如果是 MVP run：

- 不展示 9-phase timeline
- 显示：
  - `workflow unavailable`
  - `MVP run`

并明确这是产品层降级，而不是错误。

## 4. Gate / Test / Review 卡片增强

### 展示字段

- PRD Gate
- Tech Gate
- Test
- Review

### 状态视觉规则

- `APPROVED`
  - 绿色
- `PASS`
  - 绿色
- `CHANGES_REQUESTED`
  - 红色
- `FAIL`
  - 红色
- `unknown`
  - 灰黄或中性警示色

### 可读性原则

- 卡片应统一尺寸
- 标签比状态值弱一级
- 状态值本身要足够醒目
- `unknown` 不应伪装成成功或失败

## 5. Merge Readiness 解释增强

### 目标

这一块不只是显示状态，而是要解释原因。

### 展示字段

- recommendation
- canMerge
- mainClean
- worktree exists
- blockedReason
- mainDirtyFiles
- worktree path

### 解释规则

#### MERGE_READY

需要明确告诉用户为什么 ready：

- overallStatus = PASSED
- canMerge = true
- mainClean = true
- worktree exists = true

#### BLOCKED

需要明确告诉用户为什么 blocked，常见原因包括：

- dirty main
- missing worktree
- canMerge = false

### 呈现方式

推荐做一个“Ready reason / Blocked reason”说明区：

- ready 时显示正向解释
- blocked 时显示负向解释

让用户不用自己从多个字段推理。

## 6. JSON / Debug 区增强

### 目标

保留诊断价值，但不干扰普通用户阅读。

### 交互建议

- 默认折叠整个 JSON / Debug 区
- 展开后仍保留 tabs：
  - status
  - report
  - workflow

这是比“默认常驻可见”更合适的平衡。

### error 展示规则

如果存在：

- `statusError`
- `reportError`
- `workflowError`

则在对应 tab 内显示错误信息，而不是空白。

### 普通用户阅读原则

- JSON/debug 不进入首屏优先区
- 不挤占 summary / workflow / gates / merge readiness 的位置

## 空状态 / 错误状态 / 降级规则

## 空状态

### 无 runs

显示：

- `No runs found`
- 引导回 CLI 生成 run

### 某个 run detail 不可用

显示：

- `detail unavailable`
- 说明是 `status`、`report` 还是 `workflow` 缺失

## 错误状态

### report / status 局部错误

页面仍应可打开，但对应区域显示局部错误提示。

### workflowError

如果是 `NOT_FULL_WORKFLOW`：

- 视为正常降级
- 不显示为异常红错

如果是其他 workflow 错误：

- 视为局部错误

## MVP run 降级规则

MVP run 一律：

- 不渲染 9 phases
- 显示 `workflow unavailable / MVP run`
- 其他区块照常渲染

## 验收标准

V2.1 应满足：

1. sidebar 中 failed / running / merge ready 一眼可区分
2. 顶部 Summary 能直接回答：
   - overallStatus
   - recommendation
   - canResume
   - canMerge
   - mainClean
   - worktree exists
3. full workflow 9 phases 能清晰区分 completed / failed / pending
4. MVP run 仍显示 `workflow unavailable`
5. Gate / Test / Review 的 success / failure / unknown 有统一视觉规则
6. Merge Readiness 能明确解释 ready / blocked 原因
7. JSON / Debug 仍可访问，但不影响普通用户首屏阅读
8. `dashboard --limit / --run / --output` 行为不变
9. 不改变现有 JSON schema
